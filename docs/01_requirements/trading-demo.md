## Problem Statement

QuantBridge Trust Layer MVP(Sprint 1~5)로 Pine Script → 백테스트 → OHLCV 실데이터 파이프라인이 완성되었고, "Survives Reality Check" 배지를 받은 검증된 전략이 준비 상태이다. 하지만 이 전략을 실제로 돌리려면 현재 **TradingView Alert → 이메일 → 본인 수동 매매**를 거쳐야 하며, 그 결과 **신호를 놓친다**(잠/일/이동 중 신호가 와도 몰라서 진입 멈춤). 이 "수동 링크"가 검증된 전략의 기대수익을 침식하고, 결국 Trust Layer의 최종 가치(검증→실집행)가 닫히지 못한다.

Sprint 6는 Phase 3(CCXT/Kill Switch/AES-256) 진입 관문으로, **"24/7 자동 집행 + 안전장치"**가 MVP의 왕관이다.

## Demand Evidence

- **Q1 답변 C → A(validated strategy ready):** 메이커 본인이 "Survives Reality Check" 배지를 받을 준비가 확정적인 Pine 전략을 보유. 오늘 밤에 Trading 데모 버튼 있으면 눌러볼 의향 있음. → 가장 직접적인 demand evidence (founder-using-own-product, 검증된 상태).
- **Q2 답변 A (신호 놓침):** 현재 TradingView Alert → 이메일 → 수동 매매 워크플로우에서 제1 통증이 "신호 놓침"으로 확정됨. 주 3-10회 수준으로 잠/일/이동 중에 신호가 와도 모르고 지나가서 진입 멈춤. 검증된 전략의 기대수익에서 "진입 멈침 분"이 가장 크게 나가는 상실.
- **Status quo 간접 지표:** TV 커뮤니티에 "자동 실행 없으면 무슨 소용?" 반응이 넓게 퍼져 있다는 것은 2026-04-13 원본 디자인에서 검증됨(Trust Layer 프레이밍). 본인만의 고통이 아니라 구조적 고통.

> **Red flag self-check:** 이 수요 증거는 "외부 유저가 돈을 냈다"가 아니라 "메이커 본인이 당장 쓴다"이다. 솔로 프로젝트 기준으론 강한 신호지만, Sprint 7+ 수익화 단계에서 외부 유저 demand evidence 필요.

## Status Quo

Part-time crypto trader (Q3-B 프로필):
1. TradingView에서 Pine 전략이 신호 송출
2. TV Alert → webhook → 이메일로 수신
3. 이메일 본 시점 → 거래소(Bybit/Binance) 로그인 → 심볼 확인 → 수량/방향 계산 → 주문
4. 잠/일/이동 중 2-10분 지연 또는 완전 놓침
5. 실제 체결가 vs 백테스트 체결가 괴리 누적

**비용:** 주 3-10회 신호 중 ~30~50% 놓침 (본인 추정) + 체결 분 슬리피지 + 수동 실수 (수량 오입력 등). 검증된 전략의 연환산 기대수익에서 "침식분"이 상당 규모.

**Willingness to build:** 본인이 직접 쓸 인프라이므로 구축 의지 매우 높음. 다만 이미 Sprint 5까지 5주 이상 투자 → Sprint 6에서 실제 집행까지 닿지 못하면 Trust Layer 투자 회수가 지연됨.

## Target User & Narrowest Wedge

### Target User (Q3-B)

- 거래소: **Bybit (1차) + Binance (2차)** — MVP는 Bybit 단독
- 자본: **$5k ~ $20k** 범위
- 심볼 수: **BTC/ETH 포함 3~5종** 동시 운영
- 신호 빈도: **주 5~15회** (≈ 2~3/day, moderate frequency)
- 전략 수: 1~2개 (Q4-B 확정) — 스윙/윈프키테 인더 관점
- Pine Script 스킬: 보유 (개발자 본인 = 유저)

### Narrowest Wedge (Q4-B 확정)

**Bybit demo 실키 경로 + 1~2전략 + Kill Switch 3조건 + AES-256 실지 적용**

핵심 유저 플로우:
1. **전략 등록** (Sprint 3 CRUD 재사용) → Strategy ID 확보
2. **거래소 계정 등록** (신규) → Bybit demo API Key 입력 → AES-256 암호화 저장
3. **Strategy ↔ ExchangeAccount 바인딩** → 어떤 전략이 어느 계정으로 집행할지 매핑
4. **Webhook URL 생성** → 사용자에게 `https://.../v1/webhooks/{strategy_id}?token=<HMAC>` 같은 URL 제공
5. **신호 수신** → TV/외부에서 POST → payload validation → Strategy 매칭 → Celery task
6. **자동 집행** → CCXT로 Bybit demo 주문 → 결과 기록(성공/실패/체결가)
7. **Kill Switch 감시** → MDD %, 일일 손실 $, API 에러 연속 N회 중 하나 발동 시 해당 Strategy+Account 주문 차단
8. **감사 로그 + read-only FE 대시보드** → 주문 내역 + Kill Switch 상태 + 최근 에러

## Constraints

### 금지 사항 (Premise 준수)

- **demo 전용 평문 키 shortcut 금지** (Premise 4) — demo 키도 AES-256 파이프라인 통과
- **라이브(실돈) 주문 Sprint 6 범위 밖** (Premise 3) — Sprint 7+에서 별도
- **다거래소 Sprint 6 범위 밖** (Premise 3) — Bybit 단독
- **백테스트 결과와 다른 집행 분기 금지** — Strategy의 parsed Pine signal 로직이 백테스트/집행 양쪽에서 동일 경로를 타야 함 (divergence 시 "검증된 전략"의 의미 훼손)

### 필수 기술 제약

- 모든 금액/수량/수익률: `Decimal` 강제 (float 금지, Sprint 4 D8 교훈)
- 신규 datetime 컬럼: `AwareDateTime` + `TIMESTAMPTZ` (ADR-005)
- 주문 실행 Task: Celery prefork-safe lazy init (Sprint 4 D3 교훈)
- 동일 webhook payload 중복 주문 방지: `pg_advisory_xact_lock(hashtext(key))` (Sprint 5 교훈)
- CCXTProvider 확장: lifespan(HTTP) + worker_shutdown(Celery) 이중 생명주기 (Sprint 5 M3 패턴)
- OHLCV schema와 동일 방식으로 `trading` schema 격리 (namespacing)
- pytest 통합 테스트에서 CCXT 실호출 차단 — autouse fixture로 `fixture exchange provider` 강제 (Sprint 5 T26 패턴)
- timing-based 동시성 테스트 금지 — 결정적 probe 사용 (Sprint 5 T17 교훈)

### 운영 제약

- 웹훅 엔드포인트 인증: HMAC 서명 + token (Strategy별 token 발급)
- Idempotency-Key 지원 필수 (`POST /v1/orders` + `POST /v1/webhooks/*`) — Sprint 5+ 부채 해소
- Kill Switch 3조건 중 "API 에러 연속 N회"는 결정적 테스트 필요(반복 horodating 플래키 방지)
- 감사 로그 영구 보관 (금액 연관 이벤트는 90일 규칙에서 제외)

## Premises

1. **[핵심 가치]** Trading 데모의 제1 가치 = 사용자의 수동 링크(이메일 → 손가락) 제거. 경쟁 상대는 TradingView/3Commas가 아니라 **이메일 알림 + 본인 손가락 + 수면**.
2. **[MVP 왕관]** 제1 통증 = "신호 놓침" → MVP 왕관은 **24/7 자동 집행**. Kill Switch는 자동 집행의 필수 안전장치로 묶임(분리 불가).
3. **[거래소 범위]** Sprint 6 = **Bybit demo 단독**. Binance demo + 라이브 = Sprint 7+.
4. **[AES-256 파이프라인]** demo 키 저장도 AES-256 실전 경로 통과. 라이브 전환 시 "한 번 더 설계" 부채 없음.
5. **[Kill Switch 조건]** 2조건 확정 — MDD %(strategy-level), 일일 손실 $(account-level). 3번째 "API 에러 연속 N회"는 Open Q #2로 분리(operational polish vs capital risk). 심볼별 포지션 한도는 Sprint 7+.
6. **[단위 MVP 스택]** 웹훅 수신기 + 전략 매칭 + CCXT 데모 주문 + 감사 로그 + Kill Switch + AES-256. Sprint 3~5 자산(Strategy CRUD, Backtest API, CCXTProvider) 재사용 우선.

## Approaches Considered

### APPROACH A: Webhook Executor Only (Minimal)
- Summary: 웹훅 수신 → Strategy 매칭 → Celery task → Bybit demo 주문 → 로그. FE 없음.
- Effort: M (1주)
- Risk: Medium (Kill Switch 1조건만 → $5k-20k 수준에 얇음)
- Pros: 빠른 완성, 최소 diff
- Cons: Sprint 7 전환 시 Kill Switch 등급화 부채, 운영 UX 나쁨(DB 쿼리로 감사)
- Reuses: CCXTProvider, AwareDateTime, Celery prefork-safe, advisory_xact_lock, exchange_provider config flag

### APPROACH B: Full Trading Domain (Ideal) — CHOSEN
- Summary: `orders` + `exchange_accounts` + `kill_switch_events` 3테이블 신설. `ExchangeProvider` 프로토콜. Kill Switch 3조건. AES-256 실전 경로. 최소 read-only FE 대시보드.
- Effort: L (Sprint 6 꽉 참)
- Risk: Low
- Pros: Q3-B 수준에 적절한 안전망, Premise 4 준수, Sprint 7 Binance 추가가 Provider 구현 1개로 끝남, 재사용 자산 전부 활용
- Cons: Sprint 6 timebox 꽉 참, E 트랙 부채는 2개만 병행
- Reuses: A + Sprint 4 BacktestRepository 3-guard 패턴 + Strategy 도메인 + Sprint 4 SQLModel/Pydantic V2 + Alembic AwareDateTime

### APPROACH C: Inbox-First, Zero-Order MVP (Creative)
- Summary: 주문 자동화 안 함. 웹훅 → Strategy 매칭 → Telegram/Discord 봇 "원터치 실행" 알림. Kill Switch = "STOP 전략" 버튼.
- Effort: S (3~5일)
- Risk: Medium (Premise 2 위반 — Q4-B committ 되돌림)
- Pros: 최소 비용으로 신호 놓침 본질 해결, E 트랙 4~5개 병행 가능
- Cons: Sprint 7 full auto 전환 시 이원화 부채, 사용자 committ 되돌림
- Reuses: Strategy 도메인, AwareDateTime

## Recommended Approach

**Approach B: Full Trading Domain**

Q1-A(검증된 전략 준비됨) + Q3-B($5k-20k) + Q4-B(Bybit demo + Kill Switch 3조건 + AES-256) 답변이 곧 B의 구성. A는 Kill Switch가 얇아서 자본 수준 대비 위험, C는 사용자의 Q4 committ를 되돌림.

**Approach C의 통찰 흡수:** Approach B의 "감사 로그 + 주문 상태 알림" 채널로 Telegram/Discord 웹훅 **선택적** 통지를 붙이면 실시간 모니터링 UX가 공짜. plan-eng-review 단계에서 이 선택 옵션 검토 권장.

### Architecture (high-level)

#### 신규 도메인: `trading/`
- `src/trading/` — 주문 도메인 (Sprint 3 Strategy 패턴 계승)
  - `models.py` — `Order`, `ExchangeAccount`, `KillSwitchEvent` (AwareDateTime 강제)
  - `providers.py` — `ExchangeProvider` Protocol + `BybitDemoProvider` (CCXT 기반, OHLCVProvider 계보)
  - `encryption.py` — AES-256 키 관리 (cryptography.Fernet + key rotation policy)
  - `kill_switch.py` — 3-condition gate (MDD %, daily loss, API error streak) — 각 조건을 독립 evaluator로 분리
  - `webhook.py` — TV JSON 호환 payload validator + HMAC 서명 검증
  - `service.py` — OrderService (3-guard update, Sprint 4 Backtest 패턴)
  - `repository.py` — OrderRepository + ExchangeAccountRepository + KillSwitchEventRepository
  - `router.py` — REST 엔드포인트 (webhook + CRUD + read-only audit)
  - `dependencies.py` — `get_exchange_provider(config)` DI (Sprint 5 T24 패턴)

#### Config 확장
```python
# src/config.py
exchange_provider: Literal["bybit_demo", "bybit_testnet", "fixture"] = "bybit_demo"
trading_encryption_key: SecretStr  # AES-256 master key (.env.example 추가)
kill_switch_mdd_percent: Decimal = Decimal("10.0")  # strategy-level MDD limit
kill_switch_daily_loss_usd: Decimal = Decimal("500.0")  # account-level daily loss
kill_switch_api_error_streak: int = 5  # consecutive API errors before gate
```

#### DB 스키마 (`trading` schema)
```
trading.exchange_accounts
  - id (UUID)
  - user_id (FK → users.id)
  - exchange (Literal: bybit/binance)
  - mode (Literal: demo/testnet/live)
  - api_key_encrypted (TEXT, AES-256)
  - api_secret_encrypted (TEXT, AES-256)
  - created_at, updated_at (AwareDateTime)

trading.orders
  - id (UUID)
  - strategy_id (FK → strategies.id)
  - exchange_account_id (FK → trading.exchange_accounts.id)
  - symbol, side, quantity (Decimal 18,8), type
  - state (Literal: pending/submitted/filled/rejected/cancelled)
  - webhook_payload (JSONB)
  - exchange_order_id (nullable)
  - filled_price (Decimal, nullable)
  - error_message (nullable)
  - idempotency_key (UNIQUE)
  - submitted_at, filled_at (AwareDateTime)

trading.kill_switch_events
  - id (UUID)
  - trigger_type (Literal: mdd/daily_loss/api_error)
  - strategy_id or exchange_account_id (nullable, mutually exclusive)
  - trigger_value (Decimal) / threshold (Decimal)
  - triggered_at (AwareDateTime)
  - resolved_at (nullable AwareDateTime)
  - resolution_note (nullable)
```

#### API 신규 엔드포인트 (endpoints.md 업데이트)
```
POST   /v1/exchange-accounts         # 등록 (+ AES-256 암호화)
GET    /v1/exchange-accounts         # 목록 (복호화 안 함, metadata only)
DELETE /v1/exchange-accounts/{id}
POST   /v1/webhooks/{strategy_id}    # 신호 수신 (HMAC 검증)
GET    /v1/orders                    # 목록 (pagination: limit/offset — Sprint 5 M4 drift 준수)
GET    /v1/orders/{id}               # 상세
POST   /v1/orders/{id}/cancel        # 취소
GET    /v1/kill-switch/events        # 감사 로그
POST   /v1/kill-switch/resolve/{id}  # 수동 해제
```

#### Frontend read-only 대시보드
- 경로: `/trading` (기존 `/dashboard` 라우트 트리에 편입)
- 패널 3개:
  - **Orders table:** 최근 50건, state/심볼/PnL/에러 메시지
  - **Kill Switch status:** 현재 active gate + 최근 이벤트 10개
  - **Exchange accounts:** 등록된 계정 목록 + API Key masked 표시
- Sprint 6에 create/delete UX는 제외 (CLI/API로 등록)

### Pre-Sprint Gate: D0 — Baseline 데이터 수집 (병행 가능)

**Sprint 6 D1 코딩 착수 전 또는 착수와 병행** 반드시 실행 — "The Assignment" 섹션 참조.

- 본인 현재 TradingView 이메일 워크플로우의 **주당 신호 수 + 놓친 비율 + 실행 지연**을 48h 이상 측정
- 결과: `docs/01_requirements/trading-demo-baseline.md` (신호 5건 이상)
- 이 파일 없으면 Success Criteria #5("진입 멈침률 ≤ 5%") 검증 불가 — Sprint 6 배포 후 baseline 요청하면 이미 늦음

### Implementation Priority (Sprint 6 D 트랙 14 working days, parallel-friendly)

**주 1 (Day 1-7)**
1. **D1. trading schema + Alembic migration** (1.5d) — `exchange_accounts`, `orders`, `kill_switch_events` 3 테이블. `trading` schema 격리. `AwareDateTime` 강제. idempotency_key UNIQUE constraint on `orders`.
2. **D2. `ExchangeProvider` Protocol + `BybitDemoProvider` + HTTP lifespan** (2d, **parallel with D1**) — CCXT create_order/cancel_order + error 분류. FastAPI lifespan은 `src/core/lifespan.py`에서 `get_ccxt_provider_for_worker()` 재사용(Sprint 5 T22 연장). Celery worker_shutdown hook은 기존 `celery_app.py` 경로 그대로.
3. **D3. `encryption.py` AES-256** (1d, **parallel with D1/D2**) — `cryptography.Fernet` (timestamp-salted). Master key = `TRADING_ENCRYPTION_KEY` env var, **Sprint 6에선 rotation 없음**(ADR로 Sprint 7+ 이연 명시, 아래 Open Question #1 참조).
4. **D4. `OrderRepository` + `OrderService` (3-guard update 패턴)** (1.5d) — **3-guard 정의:** (1) `SELECT` with `WHERE status = old_state`, (2) `UPDATE ... WHERE id = ? AND status = old_state`, (3) `rowcount > 0` 확인 → 실패 시 race 감지. Sprint 4 `BacktestRepository.update_conditionally` 계보. idempotency_key를 SERVICE 레이어에서 hash+조회(네이티브 지원, 별도 E 트랙 아님).
5. **D5. Celery `trading.execute_order_task`** (1d) — prefork-safe lazy init (Sprint 4 D3 교훈). exchange_provider를 lazy global로.
6. **D6. webhook endpoint + HMAC + native Idempotency-Key** (1.5d) — `POST /v1/webhooks/{strategy_id}?token=<hmac_secret>` 라우트. payload JSON validator. `Idempotency-Key` header 읽어 D4 OrderService에 전달 → 중복 시 cached result 반환(201). HMAC secret은 `strategies.webhook_secret VARCHAR(64)` 신설 컬럼에 저장. rotation 엔드포인트 `POST /v1/strategies/{id}/rotate-webhook-secret` 포함.
7. **D7. Kill Switch evaluator** (1.5d) — 조건 2개 확정 + 1개 Open Q 결정:
   - **(Confirmed) MDD % per-strategy** — 각 Strategy의 `orders` 테이블에서 "진입 이후 최대 손실 / 누적 자본"이 threshold 초과
   - **(Confirmed) 일일 손실 $ per-account** — 각 ExchangeAccount의 `orders` 중 "당일(UTC) realized PnL 합"이 threshold 초과
   - **(Open Q → 아래 참조) API 에러 연속 N회** — 채택 시 evaluator 사양: `OrderService.execute()` 내 `BybitDemoProvider.create_order` 실패 발생 시 `orders` 레코드의 `error_count`를 increment, 성공 시 0으로 reset. 연속 N회 시 해당 ExchangeAccount gate. mock provider로 N회 연속 실패 deterministic 테스트 가능(timing 없음).

**주 2 (Day 8-14)**

> 이건 엔지니어링 과제가 아니라 "측정 기반 의사결정" 과제다. 숫자 없이 만든 MVP는 "내가 만들고 싶어서 만든 것"이지 "문제를 푼 것"이 아니다.
