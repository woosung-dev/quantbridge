👤 사용자단계 1 — Bybit Demo 계정 + API key 발급

외부 사이트라 Playwright 자동화 대상 아님. 사용자 본인이 진행.

  • https://demo.bybit.com 접속 (Mainnet 아님 — Demo Trading)
  • 회원가입 또는 기존 Bybit 계정으로 로그인
  • 우상단 프로필 → API 메뉴 → Create New Key → System-generated API Keys
  • Permissions:
    • ✅ Read-Write
    • ✅ Contract — Orders Position (USDT Perpetual 거래용)
    • ✅ Spot — Trade (옵션, spot 도 쓸 거면)
    • ❌ Withdraw 절대 체크 금지 (Demo 라도 습관)
  • IP whitelist: 비워두기 (개발 환경, no IP restriction)
  • 발급된 API Key + API Secret 임시 저장 (secret 은 1회만 노출)
Demo Trading 자금은 가상 USDT (10,000 USDT 기본 지급). 실제 자금 손실 없음. 단, code path 는 mainnet 과 동일 (CCXT 의 api-demo.bybit.com endpoint 만 다름).

🤖 Playwright단계 2 — ExchangeAccount 등록 (FE Dialog UI)

로그인은 사용자가 처리. dialog 열기/필드 채우기는 AI 가 Playwright 로 자동. API key/secret 본인이 직접 입력 권장 (전송 노출 방지).

  • AI 가 http://localhost:3100/trading navigate
  • Clerk 로그인 페이지 뜨면 사용자가 직접 로그인
  • 로그인 완료 후 사용자가 "다음 진행해" → AI 가 "계정 추가" 버튼 클릭
  • Dialog 열림 → AI 가 form 자동 입력:
    • 거래소: bybit
    • 모드: demo
    • 라벨: Bybit Demo Day 0 (자유)
    • API Key: (사용자가 직접 붙여넣기 권장)
    • API Secret: (사용자가 직접 붙여넣기 권장)
    • Passphrase: 해당 없음 (okx 일 때만 표시)
  • 사용자가 Submit → list 에 신규 row + api_key_masked 표시 확인
playwright 로 http://localhost:3100/trading 열어줘. clerk 로그인 페이지 뜨면 잠깐 멈춰. 내가 로그인하고 알려줄게. 그 후 "계정 추가" 버튼 누르고 dialog 에 거래소=bybit, 모드=demo, 라벨="Bybit Demo Day 0" 까지만 채워줘. api key/secret 은 내가 직접 입력할게. 
docker exec quantbridge-db psql -U quantbridge -d quantbridge \
  -c "SELECT id, exchange, mode, label, api_key_masked, created_at FROM trading.exchange_accounts;"

예상: 1 행 / exchange=bybit / mode=demo / api_key_masked 마스킹 처리 / encrypted_* 컬럼은 AES-256 암호화 binary.

🤖 Playwright단계 3 — Strategy 생성 + webhook secret 발급 (자동)

Sprint 13 의 atomic auto-issue 로 Strategy create 시 webhook_secret 도 같은 트랜잭션에 발급. plaintext 는 sessionStorage TTL 30 분 + 1 회만 노출 → 사용자가 즉시 복사 필수.

  • AI 가 http://localhost:3100/strategies/new navigate
  • AI 가 form 자동 입력:
    • 이름: EMA Cross Day 0
    • 심볼: BTC/USDT
    • Pine script: 아래 EMA crossover sample 붙여넣기 (또는 backend/tests/fixtures/pine/s1_pbr.pine 사용)
  • 사용자가 Save 클릭 → 생성 후 자동으로 strategy 상세 페이지로 이동 + ?tab=webhook 활성화
  • TabWebhook 의 amber 카드에서 Webhook URL + Secret (plaintext) 즉시 복사
  • Secret 보관 후 카드 닫기 (TTL 30 분 또는 닫는 즉시 sessionStorage 제거)
secret 못 복사하면: TabWebhook → Rotate 버튼 → ConfirmDialog → 새 secret 발급. TradingView alert 도 새 secret 으로 교체 필요.

Pine script sample (EMA crossover)

//@version=4
strategy("EMA Cross Day 0", overlay=true)

short_len = input(12, "Short EMA")
long_len  = input(26, "Long EMA")

short_ema = ema(close, short_len)
long_ema  = ema(close, long_len)

long_cond  = crossover(short_ema, long_ema)
short_cond = crossunder(short_ema, long_ema)

if long_cond
    strategy.entry("Long", strategy.long)
if short_cond
    strategy.entry("Short", strategy.short)

plot(short_ema, color=color.green)
plot(long_ema, color=color.red)
playwright 로 http://localhost:3100/strategies/new 가서 이름="EMA Cross Day 0", 심볼="BTC/USDT", pine script 는 위 EMA crossover sample 붙여넣어줘. 그리고 멈춰. 내가 Save 누르고 webhook secret 복사한 후 알려줄게. 
docker exec quantbridge-db psql -U quantbridge -d quantbridge -c "
SELECT s.id, s.name, ws.id AS secret_id, ws.created_at
FROM public.strategies s
LEFT JOIN trading.webhook_secrets ws ON ws.strategy_id = s.id
ORDER BY s.created_at DESC;"

예상: 1 strategy + 1 webhook_secret 같은 트랜잭션 (Sprint 13 atomic auto-issue). secret_id NULL 이면 broken bug 재발 — 즉시 hot-fix.

🤖 Playwright단계 4 — Backtest 1회 실행 + 결과 검증

Strategy 만들었으면 Backtest 자동화 가능. Celery 비동기 처리 → polling 5~10 초.

  • AI 가 strategy 상세 페이지 → Backtest 탭 또는 /backtests/new?strategy_id=... navigate
  • AI 가 form 자동 입력:
    • 심볼: BTC/USDT (strategy 에서 prefill)
    • 타임프레임: 1h
    • 시작일: 1 개월 전 (예: 2026-04-02)
    • 종료일: 오늘 (2026-05-02)
    • Initial capital: 10000 (USDT)
  • 사용자가 Submit → 5~10 초 polling → 결과 페이지 자동 이동
  • 5 탭 모두 렌더링 확인:
    • 📈 Equity Curve
    • 📊 Summary metrics (Sharpe / MDD / total return)
    • 📋 Trades
    • 📉 Drawdown
    • 🔢 Per-bar OHLCV
422 inline error 발생 시 (Sprint 13 Phase C): form 위 빨간 메시지 — 시작일/종료일 반드시 채울 것.
playwright 로 방금 만든 strategy 의 Backtest 탭 가서 심볼=BTC/USDT, 타임프레임=1h, 시작일=2026-04-02, 종료일=2026-05-02, initial capital=10000 입력해줘. submit 도 너가 누르고 polling 끝나면 5탭 스크린샷 찍어줘. 
docker exec quantbridge-db psql -U quantbridge -d quantbridge -c "
SELECT id, status, executed_at, summary_metrics->>'total_return' AS total_return,
       summary_metrics->>'sharpe' AS sharpe
FROM public.backtests
ORDER BY created_at DESC LIMIT 1;"

예상: status=succeeded · summary_metrics JSON 채워짐 · BL-080 fix 검증 (Celery worker prefork-safe loop).

curl -s http://localhost:8100/metrics | grep -E "^(qb_active_orders|qb_pending_alerts) "

예상: 둘 다 0 유지 (backtest 는 order 발생 없음).

👤 사용자단계 5 — TradingView alert 설정 (Day 1 시작 시)

외부 TradingView. Playwright 대상 아님. 단계 3 에서 복사한 webhook URL + secret 사용.

  • TradingView (tradingview.com) 접속 — Pro 이상 필요 (webhook 기능)
  • BTC/USDT 차트 → 동일 EMA crossover 인디케이터 적용 (또는 동일 Pine script 입력)
  • Alert create (시계 아이콘) → Condition: 인디케이터의 long/short 시그널
  • Notifications → Webhook URL 체크 → 단계 3 의 URL 붙여넣기
  • Message body: QuantBridge 의 webhook handler 가 받는 JSON 포맷 — Sprint 6 의 schema 참조: 
    {
  "strategy_id": "<단계 3 strategy id>",
  "symbol": "BTC/USDT",
  "side": "{{strategy.order.action}}",
  "quantity": "0.001"
}
  • "Send alert" → backend webhook handler 가 HMAC-SHA256 검증 → execute_order_task 발화
  • 5초~10초 후 /trading Orders panel 에 신규 order 표시 (submitted → filled)
secret 검증 (HMAC-SHA256): TradingView Pro 의 alert message body 는 plaintext. webhook handler 는 X-Signature 헤더의 HMAC 을 단계 3 secret 으로 검증. TradingView 자체는 HMAC 헤더 발송 기능 없음 → 대안: Sprint 13 의 TestOrderDialog 로 brower 에서 직접 HMAC 발사 (smoke test).
playwright 로 http://localhost:3100/trading 가서 OrdersPanel 의 "Test Order" 버튼 (NEXT_PUBLIC_ENABLE_TEST_ORDER=true 일 때만 보임) 클릭, dialog 에 strategy_id=<위 id>, side=buy, quantity=0.001 입력해줘. submit 은 내가 누를게. 그 후 Orders panel 에 새 order 뜨는지 polling 해줘.

SELECT 'exchange_accounts' AS tbl, COUNT(*) FROM trading.exchange_accounts
