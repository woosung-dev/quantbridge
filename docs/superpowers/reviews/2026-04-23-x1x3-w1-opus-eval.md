## Q8. Architecture & Maintainability — env vs per-strategy config

**Short-term: env (현재 방식) GO. Long-term: per-strategy field 권장.**

### Celery worker 동작 분석 (사실)

- QuantBridge Celery: prefork pool 고정 (CLAUDE.md "Celery prefork-safe" 규칙).
- prefork 구조: master 가 fork() 로 child worker 를 만듦. **fork 시점 부모 프로세스 환경변수가 child 로 복사됨**. 그 후 child 의 `os.environ` 는 worker 종료까지 동일 (외부에서 동적 변경 불가능, 컨테이너 재시작 필요).
- 그러나 `_get_heuristic_mode()` 가 `os.environ.get(...)` 을 매번 호출 → **동일 worker 프로세스 내 동일 값** 유지. fork 시점 캡처와 동등.
- 결과: Celery worker 풀 전체가 **단일 mode** 로 고정. 한 워커가 strict, 다른 워커가 loose 일 수 없음 (배포 단위로 균일).

### Per-strategy 전환 시 변화

- `Strategy.alert_mode: Literal["strict", "loose"]` SQLModel 필드 추가 + Alembic migration.
- `classify_message(text, mode="strict")` signature 확장 (default kwarg → 후방호환 유지 가능).
- backtest task 에서 strategy 로드 시 mode 주입 → alert hook 으로 전달.
- 장점:
  1. **사용자 단위 mix** — 한 유저가 strategy A는 strict, B는 loose 사용 가능.
  2. **A/B 비교** — 동일 strategy 를 mode 만 바꿔 백테스트 비교 (현재 env 로는 cluster restart 필요).
  3. **dogfood granularity** — i2_luxalgo 한 strategy 만 loose, 나머지 안전 모드 strict 유지.
- 단점:
  1. Alembic migration + Strategy schema/router/service 4-5 파일 변화 + FE Strategy 폼에 toggle UI.
  2. 본 W1 (244 lines) → 보수적으로 600-800 lines 로 확장.
  3. Per-strategy field 도입 시 trading session 의 alert/order 경로에도 mode 전파 필요 (Sprint 7d trading sessions 와 충돌 검토).

### Trade-off 요약

| 축             | env (W1 현재)                          | per-strategy (long-term)           |
| -------------- | -------------------------------------- | ---------------------------------- |
| Scope          | ~244 lines, 1 PR                       | ~600-800 lines, 1-2 PR + migration |
| 배포 결합      | 강 (cluster 단위 균일, 변경 시 재시작) | 약 (DB row level)                  |
| 사용자 mix     | 불가                                   | 가능                               |
| A/B 백테스트   | 불가 (rerun 시 cluster 재시작)         | 가능 (param 토글)                  |
| 회귀 위험      | 매우 낮음 (1 함수 분기)                | 중간 (schema + 4 도메인 영향)      |
| dogfood timing | 즉시 (이번 sprint)                     | H2 권장                            |

### 권장 path

1. **W1 = env mode (현재) 그대로 land** — i2_luxalgo dogfood unblock 즉시 (이번 sprint goal).
2. **`PINE_ALERT_HEURISTIC_MODE=loose` 를 모든 staging/prod 워커 기본값으로 운영 1-2주** — 회귀 데이터 수집 (loose 가 잘못된 LONG_ENTRY 를 만든 case 존재 여부).

   - Sprint 7d (Trading Sessions) 가 alert mode 다양성을 요구할 때.
