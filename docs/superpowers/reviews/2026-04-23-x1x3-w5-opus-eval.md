## Q8. Architecture & maintainability (Opus extension)

**Current shape:** `RerunButton({ backtest: BacktestDetail, isEnabled: boolean })` — accepts the entire detail payload, parses `initial_capital` internally with `Number()`.

### Pros of current design

- Always in sync with backend schema additions: if `BacktestDetail` gains a new param (e.g. `commission_rate`, `slippage_bps`), only the button needs updating; parent doesn't have to remember to pipe each new field.
- Single source of truth — parent already has `bt = detail.data`; passing the whole object avoids 6 prop names polluting the JSX (`backtest-detail-view.tsx:94-99`).
- Co-locates "what params constitute a re-run" with the re-run logic itself — strong cohesion.

### Cons

- Over-coupling to `BacktestDetail` shape: the button now depends on the _whole_ type, not just the 6 fields it cares about. If `BacktestDetail` evolves (e.g. removes/renames `initial_capital`), button breaks unnecessarily even when called from a context that has the 6 params in some other shape.
- Type-cast smell: `timeframe as never` (`rerun-button.tsx:38`) is a sign that `BacktestDetail.timeframe` (probably `string`) does not align with `BacktestCreate.timeframe` (zod enum). A purpose-built `RerunParams` interface would surface this mismatch at the boundary instead of hiding it.
- `Number(backtest.initial_capital)` placement: schema already transforms decimalString → number (`rerun-button.tsx:35-36` comment), so the `Number()` is a defensive double-cast. Fine for now; if BE schema ever changes shape, the guard is in the wrong layer (UI button) instead of at the API boundary.

### Long-term placement recommendation

Push the re-run _logic_ (param extraction + mutation) into a feature hook, keep the _UI_ in the component:

```ts
// features/backtest/hooks.ts (or rerun.ts)
export function useRerunBacktest(opts?: { onSuccess?: (id: string) => void }) {
  const create = useCreateBacktest({
    onSuccess: (created) => {
      toast.success("재실행 시작");
      opts?.onSuccess?.(created.backtest_id);
    },
    onError: (err) => toast.error(`재실행 실패: ${err.message}`),
  });
  return {
    rerun: (bt: BacktestDetail) => {
      const capital = Number(bt.initial_capital);
      if (!Number.isFinite(capital) || capital <= 0) {
        toast.error("재실행 실패: 유효하지 않은 초기 자본");
        return;
      }
      create.mutate({
        strategy_id: bt.strategy_id,
        symbol: bt.symbol,
        timeframe: bt.timeframe as BacktestCreate["timeframe"],
        period_start: bt.period_start,
        period_end: bt.period_end,
        initial_capital: capital,
      });
    },
    isPending: create.isPending,
  };
}
```

Then `RerunButton` becomes a 25-line dumb UI consuming `useRerunBacktest`, and the same logic can be reused from a list-page hover action, a keyboard shortcut, or a context menu without copy-pasting the 6-field mapping.

**Verdict:** acceptable as-is for Sprint X1+X3 (single call site, dogfood urgency wins). File a `[NICE-TO-HAVE]` follow-up to extract `useRerunBacktest` when the second caller appears (likely backtest list row action). No blocker.
