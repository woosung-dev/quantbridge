## 6. CRITICAL — Pagination disclosure accuracy ★

**검증 결과: 정확하고 informative. Misleading 아님.**

### 조건식 정확성

`trade-analysis.tsx:69-71`:

```tsx
{
  trades && num_trades > 0 && trades.length < num_trades ? (
    <p className="mt-2 text-xs text-[color:var(--text-muted)]">
      * 표시된 거래 {trades.length}건 기준 (전체 {num_trades}건 중).
    </p>
  ) : null;
}
```

- `trades` 정의됨 + `num_trades > 0` (전체 거래 ≥ 1) + `trades.length < num_trades` (현재 표시 페이지 < 전체) → 모두 만족할 때만 disclosure 표시.
- 200건 미만 backtest 또는 정확히 200건 backtest 에서는 **표시 안 됨**. 사용자가 "내 화면이 잘못됐나?" 라고 오해할 경계 케이스 없음.
- 200건 cap (`backtest-detail-view.tsx:29` `TRADE_QUERY = { limit: 200, offset: 0 }`) 와 정확히 정합.

### 문구 misleading 여부 — informational 측면

- 현재 문구: `* 표시된 거래 N건 기준 (전체 M건 중).`
- Codex round 1 → round 2 → round 3 진화 history (codex-self.md):
  - Round 1: "거래 목록 탭을 참고하세요" 추가 → Round 2 codex 가 trade-table 도 동일 200건 cap 임을 발견 (`trade-table.tsx:118` 의 "최대 200건만 표시됩니다") 하고 NO_GO 처리.
  - Round 2 fix: 탭 안내 제거, 사실만 표기 → Round 3 GO (96%).
- 결과 문구는 사용자에게 "현재 차트는 첫 200건 기준이며 이는 전체의 일부" 라는 사실만 전달. 차트가 잘못됐다는 인식보다는 한계 명시 — informative.

### Disclosure 가 "차트가 잘못" 으로 오해될 여지

- 문구가 `*` (asterisk + footnote 스타일) 로 시작 — 학술/금융 자료의 표준 disclaimer 패턴. UI 위치도 카드 group 바로 아래 (mt-2). 사용자가 "데이터 손상" 으로 오해할 시각 단서 없음.
- 단, 강한 시그널 (예: warning 색상 또는 tooltip 으로 "전체 결과를 보려면 ...") 은 부재. **현 dogfood 단계 (export 기능 부재) 에서는 적정**, 향후 UX 강화 여지는 있으나 W4 scope 외.

**결론:** 정확하고 misleading 하지 않음. Codex 의 2회 iteration 으로 disclosure UX 가 보수적으로 다듬어진 흔적 양호.

## 8. Architecture & maintainability — Client-side vs Server-side breakdown ★

### 단기 (현재 W4 그대로)

**유지 권장.** 이유:

- BE schema 변경 없이 dogfood 빠른 반복 가능.
- 200건 cap 은 plan §1 에서 인지된 trade-off 이며 disclosure 로 사용자에게 사실 노출.
- s3_rsid / 코퍼스 기준 대부분 backtest 가 50-150 trade 수준 → 현재 대다수 케이스는 cap 미히트.

### 장기 (H2 또는 dogfood 데이터 누적 후)

**BE 통합 권장.** 권장 schema 확장:

```python
# backend/src/backtest/schemas.py BacktestMetricsOut 추가:
long_count: int               # 이미 존재
long_win_count: int            # 신규
long_total_pnl: Decimal        # 신규
short_count: int               # 이미 존재
short_win_count: int           # 신규
short_total_pnl: Decimal       # 신규
```

FE 는 `metrics.long_total_pnl / metrics.long_count` 로 평균 계산 (Decimal precision 보존). 현 client-side util 은 deprecate 하지 않고 fallback 으로 유지 (`metrics.long_win_count` 부재 시 trades 기반 계산 → 구 backtest 호환).

### 마이그레이션 경로

1. **Phase A (W4 현재):** client-side compute, disclosure 표시. ✅
2. **Phase B (BE schema 확장 + alembic migration):** `BacktestMetricsOut` 에 6개 필드 추가, vectorbt 결과 wiring. computer-side 로직은 지표 계산 시점 (Celery task) 에 1회만 실행 → 정확도 + persistance.
3. **Phase C (FE refactor):** `BacktestMetricsOut.long_win_count != null` 체크 후 server-side 우선, 없으면 client-side fallback. 점진적 cutover.

### Trade-off 요약

| 측면              | Client (현재 W4)         | Server (제안)                             |
| ----------------- | ------------------------ | ----------------------------------------- |
| 정확도            | 200건 cap                | 100%                                      |
| 구현 속도         | 즉시 (W4 완료)           | BE schema + migration + FE wiring 필요    |
| Decimal precision | float 변환 손실 가능     | Decimal 유지                              |
| 캐싱              | 매 페이지 로드 시 재계산 | 1회 계산 후 DB persist                    |
| 회귀 위험         | 0 (BE 미변경)            | alembic migration + 구 backtest 호환 처리 |
| Dogfood iteration | 빠름                     | 1 sprint 추가                             |

**최종 권장:** 현재 W4 GO + H2 (Phase B/C) BE 통합 follow-up sprint 등록 (`docs/TODO.md` 추가 권장 — 별도 ticket).
