## Q2 — Spurious PASS: reversal/AF cap 테스트 엄밀성

### Reversal 테스트 (`test_stdlib_sar.py:69-83`)


**문제점:** 두 assertion 모두 약한 검증.

- `sar[6] > sar[4]`: SAR가 단순히 "올라갔다"만 확인. EP=110에 정확히 reset됐는지 검증 안 함.

**spurious PASS 가능성:** 만약 reversal 로직이 새 SAR = `prev_ep + ε`처럼 잘못 구현돼도 (예: `state.sar = prev_ep * 1.001`) 두 assertion 모두 통과 가능. EP 부정확한 값 (예: 109.5)도 통과.

## Q8 (Opus) — Architecture & maintainability

**현재 패턴 (hybrid):**

- `SarState` `@dataclass` (`stdlib.py:31-49`) — 9개 필드 type hint 명시
- 저장 위치 `state.buffers[node_id]` dict slot (`stdlib.py:576` `setdefault(node_id, SarState())`)
- 다른 stdlib state는 모두 raw dict slot (`{"prev": nan, "warmup": []}` 등 `stdlib.py:69,120,158,216,240,268,298`)

### 단기 (이번 sprint ~ Sprint 8d/X1+X3 종료)

**현 패턴 유지 권장.** 이유:

1. **Dispatcher parity 우선** — `state.buffers[node_id]` slot은 모든 ta.\* state-holder의 공통 contract. SarState만 별도 컬렉션으로 빼면 dispatcher branch가 비대칭이 되고, runtime reset (e.g. `state.buffers.clear()` 시 SAR만 살아남는 버그) 위험.
2. **Type clarity 부분 회수 충분** — dataclass로 9개 필드 명명하니 raw `{"is_uptrend": bool, ...}` 보다 IDE 추론·refactor 안전. ta.atr/ta.rsi는 3-4 필드라 dict가 적정. SarState는 9 필드라 dataclass가 적정. **상태 복잡도에 비례한 자연스러운 분기.**
3. **Visitor pattern은 over-engineering** — 현재 dispatcher는 if-else 14개 분기로 단순. Visitor (RsiVisitor, AtrVisitor, SarVisitor 클래스 14개) 도입은 cognitive overhead 증가하고 "1 line 추가"가 "1 클래스 + 1 분기 + 1 register" 3 line으로 늘어남.

### 장기 (H2 pine_v2 심화 — local history ring, valuewhen cap, user fn state isolation)

**typed state subclass 도입 시점은 다음 중 하나일 때:**

(a) **stdlib state-holder ≥ 20개 + 평균 필드 ≥ 5** — dict 기반 추론 비용이 dataclass 비용 초과
(b) **runtime introspection 필요** — debugger/profiler가 state 종류별 inspect (e.g. "이 strategy가 점유한 SAR state 수")
(c) **state versioning / migration** — 백테스트 결과 직렬화 시 state schema가 추적돼야 함 (현재는 vectorbt가 final result만 저장)

**권장 path (장기):**

```python
# Step 1 (H2 초): protocol 정의
class StdlibState(Protocol):
    """모든 ta.* state-holder 공통 인터페이스"""
    def reset(self) -> None: ...

# Step 2 (H2 중): 복잡도 ≥ 5 필드 state는 dataclass + StdlibState 구현
@dataclass
class SarState:  # 이미 9 필드
    ...
    def reset(self) -> None:
        self.is_initialized = False
        ...

# Step 3 (H2 말): dispatcher가 state.buffers[node_id]: StdlibState 추론
sar_state: SarState = self.state.buffers.setdefault(node_id, SarState())
return ta_sar(sar_state, ...)
```

이 경로는 **dispatcher 패턴은 유지**하면서 **타입만 점진 강화** — visitor pattern 우회로 cognitive cost 최소.

### 핵심 권장

| 시점             | 행동                                                                | 근거                                                     |
| ---------------- | ------------------------------------------------------------------- | -------------------------------------------------------- |
| 현재 (PR #X1+X3) | **하이브리드 유지**                                                 | dispatcher parity, complexity 적정                       |
| H2 초            | `StdlibState` Protocol + 복잡도 ≥ 5 필드 state는 dataclass          | type clarity 점진 강화                                   |
| H2 말 / 필요 시  | runtime introspection이 명시 요구되면 typed subclass + visitor 검토 | over-engineering 회피                                    |
| **하지 말 것**   | 지금 visitor pattern 14개 클래스 도입                               | premature abstraction, dispatcher 1줄 추가의 단순함 손상 |

**판정:** Worker의 hybrid 선택은 **상태 복잡도(9 필드)에 정확히 맞는 적정 추상화.** Long-term Protocol+dataclass migration은 H2 stdlib 확장 (local history ring 등) 동반 시 자연스럽게 도입 권장.
