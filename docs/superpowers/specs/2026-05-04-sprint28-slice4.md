### 결정 1: capital_base fetch timing (ADR-006 결의 사항)

**현재 상태** (Explore 보고서 §A, B):

- `backend/src/services/trading/kill_switch.py` (Sprint 8 hybrid 모드) — `CumulativeLossEvaluator` 가 `balance_provider` optional 받음
- `backend/src/services/exchange/service.py:127-157` — `fetch_balance_usdt(account_id)` (cache 없음, 매 호출 ~200ms)
- ADR-006 [미해결]: **capital_base fetch timing 의사결정**

**Option A:** KillSwitch trigger 시 매번 호출 (정확도 최대, latency +200ms)
**Option B:** 5초 TTL cache (latency 0, 5초 stale data)

**Explore 보고서 권장:** **Option A** — Beta Path A1 = "capital safety 우선" + Cross Margin effective_leverage 변동 가능 + fetch_balance 실패 시 fallback config (10k USD) 으로 충분한 안전장치.

**결정:** **Option A** (KillSwitch trigger 시 매번). T1 에서 ADR-006 정식 결의.

### 결정 2: Cross Margin effective leverage cap

**현재 상태** (Explore §A, E):

- `backend/src/services/trading/order_service.py:312-317` — `settings.bybit_futures_max_leverage` (20x) 초과 시 422

- **누락:** `fetch_positions()` 합산 — 기존 open position 무시. effective_leverage 계산 부정확.

**공식:** `effective_leverage = sum(open_position_value_usdt) / total_equity_usdt`

**Option A:** `OrderService._execute_inner()` 안 fetch_positions() 호출 + 합산 추가
**Option B:** KillSwitch evaluator 안 별도 `EffectiveLeverageEvaluator` 신규
**Option C:** 본 Slice 4 scope 외 (선택사항) — Slice 4 = fetch_balance timing 결의만

**결정:** **Option B** (`EffectiveLeverageEvaluator` 신규) — KillSwitch evaluator pattern 정합 + OrderService 와 분리. 단, **시간 부족 시 Slice 4 에서 deferred (Slice 4 scope 명확)** + 별도 BL 등록.
