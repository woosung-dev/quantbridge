### 결정 2: Celery prefork-safe 패턴 적용

**영구 규칙** (Sprint 4 D3, `.ai/stacks/fastapi/backend.md:§11`):

- Engine + sessionmaker는 **per-task 생성 + finally dispose**
- module-level async state 금지

**적용:**

```python
# backend/src/tasks/market_data_backfill.py
@celery_app.task(name="market_data.backfill_ohlcv", max_retries=0)
def backfill_ohlcv_task(symbol: str, timeframe: str, period_days: int) -> dict:
    from src.tasks._worker_loop import run_in_worker_loop
    return run_in_worker_loop(_async_backfill(symbol, timeframe, period_days))

async def _async_backfill(...):
    engine, sm = create_worker_engine_and_sm()
    try:
        async with sm() as session:
            repo = OHLCVRepository(session)
            ccxt = CCXTProvider(...)
            provider = TimescaleProvider(repo, ccxt)
            return await provider.backfill(symbol, timeframe, period_days)
    finally:
        await engine.dispose()
```

### 결정 3: backfill batch size + rate limit

**Bybit Demo rate limit:** ~200 requests/min = 333ms/req (보수적: 500ms/req)
**CCXT batch limit:** 500 candles per request

**계산:**

- BTC/USDT 1H × 30일 = 720 candles = 2 requests
- BTC/USDT 5m × 30일 = 8640 candles = 18 requests × 500ms = 9s
- BTC/USDT 1m × 30일 = 43200 candles = 87 requests × 500ms = 44s
- ETH/USDT + SOL/USDT 동일 = 3배

**결정:** 1차 backfill = BTC/USDT (1H + 5m + 15m + 1h) ≥30일 ≈ 60s. ETH/SOL 후속 sprint 또는 user-triggered.

### 결정 4: race condition 회피 (backfill ↔ LiveSession dispatch)

**위험:** backfill task 가 large fetch 중 동일 symbol 의 LiveSession dispatch 가 advisory lock 대기 → dispatch 지연 → market 변동 mismatch

**Option A:** Lock timeout 5분 + retry (LiveSession dispatch 가 lock 못 얻으면 다음 주기)
**Option B:** backfill task 가 lock 후 chunk 분할 (10분 단위) 실행 → LiveSession 사이사이 가능
**Option C:** backfill 시간 (dogfood 휴식 시간 = KST 03:00) 으로 제한

**결정:** **Option A** + **Option C** — lock 5분 timeout + 사용자 manual trigger (KST 03:00 권고). LiveSession dispatch 는 다음 주기에서 정상 진행.
