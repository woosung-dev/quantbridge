## Brainstorming (의무 45분, 5 핵심 결정)

### 결정 1: equity_curve schema 위치

**현재 상태** (Explore 보고서 §B):

- `backend/alembic/versions/20260504_0001_add_live_signal.py:155-194` — `trading.live_signal_states` 이미 `total_realized_pnl` (NUMERIC 18,8) 보유

**Option A:** `live_signal_states.equity_curve` JSONB array `[{timestamp_ms, cumulative_pnl: "0.123"}]`

- Pros: UI 스트리밍 친화적, single table query
- Cons: JSONB array growth (cumulative — long-running session 시 1000+ entry)

**Option B:** 별도 `trading.live_signal_equity_points` timeseries 테이블

- Pros: 분석 쿼리 최적화, retention/aggregation 가능
- Cons: write 빈도 ↑, JOIN 필요, scope ↑

**Option C:** No persistence — on-demand 계산 (LiveSignal events 에서 매 fetch 시 재계산)

- Pros: write 0, schema 변경 0
- Cons: read latency ↑ (특히 long session)

**결정:** **Option A** (JSONB) — Sprint 28 scope 단순화 + realtime streaming 친화적. Long-running session 시 JSONB compaction (예: 1시간 단위 aggregation) 은 후속 BL.

### 결정 2: equity 계산 시점

**Option A:** 매 event 시점 (open/close 마다 누적 갱신) — 가장 정확
**Option B:** 주기적 cron (5분 마다 active LiveSignal 갱신) — write 빈도 ↓
**Option C:** on-demand (DB 저장 X, fetch 시 계산) — write 0

**Explore 보고서 권장:** Option A (매 event) — 사유: `evaluate_live_signals_task` (1분 Beat) 안 transaction 에 누적 write 가능, bar 당 최대 2-3 event = write 빈도 관리 가능, realtime chart 요구사항 충족.

**결정:** **Option A** + JSONB compaction 후속 BL 등록.

### 결정 3: Decimal 합산 영구 규칙 적용

**Sprint 4 D8 영구 규칙:** `Decimal(str(a)) + Decimal(str(b))` (NOT `Decimal(str(a + b))`)

**기존 사례** (Explore §D):

- `backend/src/backtest/engine/trades.py:92-95` — `fees_total = Decimal(str(row["Entry Fees"])) + Decimal(str(row["Exit Fees"]))`

**equity_calculator 적용:**

```python
def accumulate_equity(prev_curve: list[EquityPoint], pnl: Decimal, timestamp_ms: int) -> list[EquityPoint]:
    last_cumulative = Decimal(str(prev_curve[-1].cumulative_pnl)) if prev_curve else Decimal("0")
    new_cumulative = last_cumulative + Decimal(str(pnl))  # ✅ 영구 규칙 정합
    return prev_curve + [EquityPoint(timestamp_ms=timestamp_ms, cumulative_pnl=str(new_cumulative))]
```
