## 2. 5개 기술 결정

### 2.1 ExchangeProvider — Per-account ephemeral CCXT client

**결정:** 주문 집행용 CCXT 클라이언트는 **ExchangeAccount마다 주문 시점에 새로 생성**하고, 주문 종료 시 즉시 `close()`. Sprint 5의 public `CCXTProvider`(OHLCV)와는 완전히 분리된다.

```python
# backend/src/trading/providers.py
from typing import Protocol
import ccxt.async_support as ccxt_async

class ExchangeProvider(Protocol):
    async def create_order(self, creds: Credentials, order: OrderSubmit) -> OrderReceipt: ...
    async def cancel_order(self, creds: Credentials, exchange_order_id: str) -> None: ...
    async def fetch_order(self, creds: Credentials, exchange_order_id: str) -> OrderStatus: ...


class BybitDemoProvider:
    async def create_order(self, creds: Credentials, order: OrderSubmit) -> OrderReceipt:
        exchange = ccxt_async.bybit({
            "apiKey": creds.api_key,
            "secret": creds.api_secret,
            "enableRateLimit": True,
            "options": {"defaultType": "spot", "testnet": True},
        })
        try:
            result = await exchange.create_order(
                order.symbol, order.type, order.side,
                float(order.quantity), float(order.price) if order.price else None,
            )
            return OrderReceipt(exchange_order_id=result["id"], filled_price=Decimal(str(result.get("average") or 0)))
        finally:
            await exchange.close()


class FixtureExchangeProvider:
    """exchange_provider=fixture 시 사용. Sprint 5 T26 autouse 패턴으로 CCXT 실호출 차단."""
    # 결정적 mock — pytest-asyncio에서 주입
```

**근거:**
- 보안: 복호화된 API Key/Secret의 수명을 함수 스코프로 한정 (Premise 4: demo도 실전 경로)
- 멀티 계정 자연 지원 (여러 ExchangeAccount 각자 credentials)
- Sprint 5 CCXTProvider(public fetch, lifespan singleton)와 책임 분리 — 공개 시장 데이터는 auth 불필요
- Sprint 7 라이브 전환 시 latency 병목 발견되면 per-account client pool(Option B)로 이행 가능. Protocol은 불변이므로 소비자(OrderService) 영향 없음

**트레이드오프:**
- 주문당 CCXT 객체 생성 ~50ms 오버헤드. 현 타깃 주 5–15 신호 프로파일에서 수용 가능
- connection pool 재사용 불가 → rate limiter state도 매번 초기화

### 2.2 Kill Switch — 별도 `KillSwitchService` + `KillSwitchEvaluator` Protocol

**결정:** MDD/일일손실(+미래 API error streak) 평가 로직을 각각 독립 `Evaluator`로 분리하고, `KillSwitchService`가 순회 평가. `OrderService`는 `ensure_not_gated(strategy_id, account_id)`만 호출.

```python
# backend/src/trading/kill_switch.py
from typing import Protocol

@dataclass
class EvaluationContext:
    strategy_id: UUID
    account_id: UUID
    now: datetime  # 주입 — 테스트에서 제어

@dataclass
class EvaluationResult:
    gated: bool
    trigger_type: Literal["mdd", "daily_loss", "api_error"] | None
    trigger_value: Decimal | None
    threshold: Decimal | None

class KillSwitchEvaluator(Protocol):
    async def evaluate(self, ctx: EvaluationContext) -> EvaluationResult: ...


class MddEvaluator:
    def __init__(self, repo: OrderRepository, threshold: Decimal): ...
    async def evaluate(self, ctx) -> EvaluationResult:
        # Strategy의 orders에서 "진입 이후 최대 손실 / 누적 자본" 계산
        ...


class DailyLossEvaluator:
    def __init__(self, repo: OrderRepository, threshold_usd: Decimal): ...
    async def evaluate(self, ctx) -> EvaluationResult:
        # ExchangeAccount의 UTC 당일 realized PnL 합
        ...


class KillSwitchService:
    def __init__(
        self,
        evaluators: Sequence[KillSwitchEvaluator],
        events_repo: KillSwitchEventRepository,
    ): ...

    async def ensure_not_gated(self, strategy_id: UUID, account_id: UUID) -> None:
        # 1. 기존 unresolved KillSwitchEvent 있으면 즉시 raise.
        #    trigger_type별 매칭 규칙:
        #      mdd → strategy_id 매칭
        #      daily_loss, api_error → account_id 매칭
        #    → repo 쿼리: WHERE resolved_at IS NULL AND (
        #         (trigger_type='mdd' AND strategy_id=:sid)
        #         OR (trigger_type IN ('daily_loss','api_error') AND account_id=:aid)
        #       )
        active = await self._events_repo.get_active(strategy_id, account_id)
        if active:
            raise KillSwitchActive(active)

        # 2. 모든 evaluator 순회 — 첫 위반에서 이벤트 기록 + raise
        ctx = EvaluationContext(strategy_id, account_id, now=datetime.now(UTC))
        for evaluator in self._evaluators:
            result = await evaluator.evaluate(ctx)
            if result.gated:
                event = await self._events_repo.create(KillSwitchEvent(
                    trigger_type=result.trigger_type,
                    strategy_id=strategy_id if result.trigger_type == "mdd" else None,
                    account_id=account_id if result.trigger_type in ("daily_loss", "api_error") else None,
                    trigger_value=result.trigger_value,
                    threshold=result.threshold,
                ))
                raise KillSwitchActive(event)
```

**OrderService 통합:**
```python
class OrderService:
    def __init__(self, repo, kill_switch: KillSwitchService, provider, account_service): ...

    async def execute(self, order_req: OrderRequest, idem_key: str) -> OrderResponse:
        await self._kill_switch.ensure_not_gated(order_req.strategy_id, order_req.account_id)
        # ... Q5 idempotency flow
```

**근거:**
- 각 evaluator 단독 테스트 — 결정적 probe(Sprint 5 T17 교훈, timing 기반 금지)
- Open Q #2(Parent) "API error streak" 채택 시 `ApiErrorStreakEvaluator` 1개 추가로 끝. `OrderService` 무변경
- 수동 주문(`POST /v1/orders`)과 웹훅 주문 양쪽에 동일 gate 자동 적용
- `KillSwitchEvent` 기록 책임이 한 서비스에 응집 — 감사 로그 누락 방지

### 2.3 AES-256 — 전용 `EncryptionService` + DI (복호화는 명시적 메서드)

**결정:** `EncryptionService`는 Fernet wrapper 단독 클래스. `ExchangeAccountRepository`는 암호문만 다루고, 복호화는 `ExchangeAccountService.get_credentials_for_order(account_id)`에서만 허용. TypeDecorator 패턴은 **사용하지 않음**.

```python
# backend/src/trading/encryption.py
from cryptography.fernet import Fernet, InvalidToken
from pydantic import SecretStr

class EncryptionService:
    def __init__(self, master_key: SecretStr) -> None:
        self._fernet = Fernet(master_key.get_secret_value().encode())

    def encrypt(self, plaintext: str) -> bytes:
        return self._fernet.encrypt(plaintext.encode())

    def decrypt(self, ciphertext: bytes) -> str:
        try:
            return self._fernet.decrypt(ciphertext).decode()
        except InvalidToken as e:
            raise EncryptionError("AES-256 복호화 실패") from e
```

```python
# backend/src/trading/service.py
class ExchangeAccountService:
    def __init__(self, repo, crypto: EncryptionService, audit_logger): ...

    async def register(self, user_id: UUID, request: RegisterAccountRequest) -> ExchangeAccount:
        account = ExchangeAccount(
            user_id=user_id,
            exchange=request.exchange,
            mode=request.mode,
            api_key_encrypted=self._crypto.encrypt(request.api_key),
            api_secret_encrypted=self._crypto.encrypt(request.api_secret),
        )
        return await self._repo.save(account)

    async def get_credentials_for_order(self, account_id: UUID) -> Credentials:
        """Provider가 주문 직전에만 호출. 여기서 audit 로깅."""
        account = await self._repo.get_by_id(account_id)
        if not account:
            raise AccountNotFound(account_id)
        self._audit.record_decryption(account_id=account_id, purpose="order_execution")
        return Credentials(
            api_key=self._crypto.decrypt(account.api_key_encrypted),
            api_secret=self._crypto.decrypt(account.api_secret_encrypted),
        )
```

**근거:**
- 복호화 = 명시적 메서드 호출 → 감사 로깅 삽입 포인트 고정 (TypeDecorator는 ORM 차원에서 투명 복호화 → 감사 어렵고, `repo.list()` 호출만으로 모든 secret이 메모리 상주)
- Q2.1 Provider 패턴과 결합: 주문 함수 스코프 내에서만 평문 노출
- Sprint 7 key rotation: `EncryptionService.rotate_all(old_key, new_key)` 배치 메서드 추가로 격리 (Fernet은 multi-key 네이티브 지원)

**트레이드오프:**
- 파일 1개·DI 체인 1단 추가 (허용)
- `get_credentials_for_order` 호출 누락 시 Provider가 복호화 못 받음 → 타입 시그너처로 강제(`Credentials` 미수신 시 타입 에러)

### 2.4 Webhook secret — `trading.webhook_secrets` 별도 테이블 (Parent 원안 A에서 변경)

**결정:** Parent는 `strategies.webhook_secret VARCHAR(64)` 컬럼을 제안했으나, **별도 테이블로 분리**. Grace period rotation을 지원하여 TV Alert drop 0건.

```sql
-- alembic migration
CREATE TABLE trading.webhook_secrets (
    id UUID PRIMARY KEY,
    strategy_id UUID NOT NULL REFERENCES strategies(id) ON DELETE CASCADE,
    secret TEXT NOT NULL,               -- HMAC 키 (추가 암호화 여부는 §6 Open Item 참조)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    revoked_at TIMESTAMPTZ NULL
);
CREATE INDEX idx_webhook_secrets_strategy_active ON trading.webhook_secrets (strategy_id, revoked_at);
```

**Rotation 엔드포인트:**
```
POST /v1/strategies/{strategy_id}/rotate-webhook-secret?grace_period_seconds=3600
```
1. 신규 row insert
2. 기존 active row(들) `revoked_at = NOW()`
3. `grace_period_seconds` 안에 들어온 구 secret HMAC도 허용

**WebhookService 검증 쿼리:**
```python
async def verify_hmac(self, strategy_id: UUID, token: str, now: datetime, grace: timedelta) -> bool:
    # revoked_at IS NULL OR revoked_at > now - grace
    candidates = await self._repo.list_valid_secrets(strategy_id, grace_cutoff=now - grace)
    return any(hmac.compare_digest(token, _sign(secret, payload)) for secret in candidates)
```

**근거:**
- `strategies` 테이블이 `trading` 도메인 관심사를 갖지 않도록 도메인 경계 유지
- Grace period → rotation 중 TV Alert 유실 0건 (Parent의 "24/7 자동 집행" MVP 왕관과 정합)
- 감사 자연 확보: 언제 rotate됐는지, 어떤 secret이 몇 건 신호 받았는지(Sprint 7+에 `orders.webhook_secret_id` FK 추가로 확장)

**트레이드오프:**
- 테이블 1개·JOIN 1단 추가 (Sprint 6 timebox 영향 < 0.5d — D6 범위 흡수 가능)
- Parent section "Architecture / DB 스키마" + "해결된 질문" 업데이트 필요 (아래 §7)

### 2.5 Idempotency-Key — DB UNIQUE + `pg_advisory_xact_lock`

**결정:** `orders.idempotency_key TEXT UNIQUE NULL`. 검사+삽입을 `pg_advisory_xact_lock(hashtext(key))`로 감싸 race 제거. Redis 의존 **없음**.

```python
class OrderService:
    async def execute(self, req: OrderRequest, idem_key: str | None) -> OrderResponse:
        await self._kill_switch.ensure_not_gated(req.strategy_id, req.account_id)

        created_order_id: UUID | None = None
        cached_response: OrderResponse | None = None

        # 1. 트랜잭션 scope: advisory lock → 중복 검사 → INSERT 까지만.
        #    Celery .delay() 호출은 반드시 commit 이후 (visibility race 방지).
        async with self._session.begin():
            if idem_key is not None:
                await self._session.execute(
                    text("SELECT pg_advisory_xact_lock(hashtext(:k))"),
                    {"k": idem_key},
                )
                existing = await self._repo.get_by_idempotency_key(idem_key)
                if existing:
                    cached_response = OrderResponse.from_order(existing)
                    # 블록 exit 시 commit, 하지만 dispatch는 하지 않음
                else:
                    order = await self._repo.create(Order(
                        idempotency_key=idem_key,
                        state=OrderState.PENDING,
                        ...
                    ))
                    created_order_id = order.id
            else:
                order = await self._repo.create(Order(
                    state=OrderState.PENDING, idempotency_key=None, ...
                ))
                created_order_id = order.id
        # commit 완료 (lock 해제, row visible).

        if cached_response is not None:
            return cached_response

        # 2. 커밋 후에만 dispatch — worker가 order row를 읽을 수 있음이 보장됨.
        assert created_order_id is not None
        await self._dispatcher.dispatch_order_execution(created_order_id)
        return await self._repo.get_response_by_id(created_order_id)
```

**핵심 불변식:** `celery.execute_order_task.delay(order_id)`는 `orders` row가 commit된 **후**에만 호출한다. `session.begin()` 블록 안에서 dispatch하면 worker가 commit 전에 조회하여 `OrderNotFound` 발생.

**근거:**
- Sprint 5 M2 TimescaleProvider와 동일 패턴 — 운영·테스트 노하우(예: `pg_try_advisory_xact_lock` 결정적 probe) 공유
- DB UNIQUE는 advisory lock race 통과 후 safety net (lock 버그 시에도 중복 주문 0건)
- 단일 store(PG)로 장애/복구 단일 경로
- Sprint 6 프로파일(주 5–15 신호)에 Redis SETNX는 과설계

**테스트 플랜:**
- `pytest-asyncio` + `asyncio.gather(req1, req2)` 동일 key → 정확히 1건 INSERT, 두 번째는 cached 응답
- 결정적 검증: 첫 요청의 트랜잭션 안에서 `pg_try_advisory_xact_lock` probe 실패 확인 (timing sleep 금지, Sprint 5 T17)

## 5. 핵심 플로우 (end-to-end)

```
POST /v1/webhooks/{strategy_id}?token=<hmac_secret>
Headers: Idempotency-Key: K
Body: TV JSON payload

  ↓ WebhookService
    (a) WebhookSecretRepository.list_valid_secrets(strategy_id, grace)  ← Q4 grace rotation
    (b) HMAC 검증(후보 secret 전부 시도, compare_digest)
    (c) Strategy load + TV payload validation
  ↓
OrderService.execute(order_req, idem_key=K):
    (d) KillSwitchService.ensure_not_gated(sid, aid)                    ← Q2
          MddEvaluator → DailyLossEvaluator → (Option: ApiErrorStreak)
    (e) pg_advisory_xact_lock(hashtext(K))                              ← Q5
    (f) get_by_idempotency_key(K) → hit 시 cached (201, same order_id)
    (g) miss: INSERT orders(state=pending, idempotency_key=K)
    (h) commit (lock 자동 해제) → Celery execute_order_task.delay(order_id)
  ↓
HTTP 201 {order_id, state: "pending"}

[Celery 비동기]
execute_order_task(order_id):
    (i)  OrderRepository.transition_to_submitted(order_id)               ← Sprint 4 3-guard
    (j)  ExchangeAccountService.get_credentials_for_order(account_id)    ← Q3 명시적 복호화
    (k)  BybitDemoProvider.create_order(creds, order):                   ← Q1 ephemeral
            ccxt.async_support.bybit({apiKey, secret}) 생성
            exchange.create_order(...)
            finally: await exchange.close()
    (l)  OrderRepository.transition_to_filled/rejected(order_id, exchange_order_id, filled_price)
```
