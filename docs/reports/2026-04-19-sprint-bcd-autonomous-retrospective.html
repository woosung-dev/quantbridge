4. 중간에 막혔던 부분

Blocker #1 · --dangerously-skip-permissions 단독 호출이 no-op

증상. cmux 탭에서 claude --dangerously-skip-permissions "..." 로 워커를 띄웠을 때, 첫 Bash 명령 echo running > .../signals/x.status 에서 퍼미션 프롬프트 1/2/3 선택 화면이 뜨면서 멈춤. 3 워커 전부 동일 증상.

원인. claude CLI 2.1.114 에서 퍼미션 바이패스는 게이트 플래그 + 실행 플래그 이중 구조로 바뀐 것으로 추정.

  • --allow-dangerously-skip-permissions — "바이패스를 옵션으로 노출" 한다는 게이트
  • --dangerously-skip-permissions — 그 옵션을 실제 적용 한다는 실행 플래그

게이트 없이 실행 플래그만 쓰면 사실상 무시됨.

조치. 두 플래그를 함께 + cmux 워크스페이스 트러스트 다이얼로그 자동 수락.

cmux new-workspace --cwd <worktree> --command \
  'claude --allow-dangerously-skip-permissions --dangerously-skip-permissions "<prompt>"'

# 10초 뒤 "이 폴더를 신뢰하시겠습니까?" 다이얼로그에 1 전송
cmux send     --workspace <ws> "1"
cmux send-key --workspace <ws> "Return"

상태 바에 ⏵⏵ bypass permissions on 이 보여야 성공. 재시도 후 B 는 00:03:37, C/D 는 00:04:18 에 정상 running 진입.

Blocker #2 · .claude/settings.json 의 gh pr merge* DENY

증상. 오케스트레이터가 gh pr merge 27 --squash --delete-branch 시도 → Claude 도구 레이어에서 프롬프트도 없이 즉시 거부. --dangerously-skip-permissions 로도 override 불가.

원인. 사용자가 직접 작성한 프로젝트 공유 baseline 가드 (deny 배열에 rm -rf, git reset --hard, git push --force, sudo, docker prune 등과 동일 범주). Git Safety Protocol 의 스피릿을 실제 강제하는 장치.

// .claude/settings.json:50-62
"deny": [
  "Bash(rm -rf*)",
  "Bash(rm -fr*)",
  "Bash(git reset --hard*)",
  "Bash(git push --force*)",
  "Bash(git push -f*)",
  "Bash(gh pr merge*)",  ← 56번째 줄
  "Bash(docker rm*)",
  "Bash(docker volume rm*)",
  "Bash(docker system prune*)",
  "Bash(docker image prune*)",
  "Bash(sudo*)"
]

조치. 사용자에게 A/B/C/D 옵션을 제시 → Option A 선택 (워커는 pr_ready 까지만, 머지는 수동). 이후 머지 시점에 사용자가 "자동" 승인 → 56번 줄 한 줄만 임시 제거 → 3 머지 수행 → 즉시 원복. 다른 deny 줄 (rm -rf, git reset --hard 등) 은 건드리지 않음.

세션 내 정확한 flow: Edit (line 제거) → gh pr merge × 3 → Edit (line 복원) → grep "gh pr merge" settings.json 로 복원 검증.

Blocker #3 · 워크스페이스 트러스트 다이얼로그 + "sensitive file" 재확인

증상. 바이패스 조합을 쓴 후에도 claude 첫 실행 시 "이 폴더를 신뢰하시겠습니까?" 가 뜸 (-p 모드에서만 자동 스킵됨). 게다가 C/D 일부 Bash 는 .claude/plans/.../signals/x.status 를 "sensitive file" 로 판정하여 두 번째 추가 확인 출현.

조치. kickoff 후 10초 대기 → cmux send "1" + Return 으로 트러스트 수락. sensitive-file 재확인에는 "2" (always allow signals/) 로 persist 승인 → 동일 범주 반복 프롬프트 제거.

교훈. 바이패스 플래그 ≠ 완전 자율. 트러스트 다이얼로그와 sensitive-path 가 별개 게이트로 남아 있음.

Blocker #4 · cmux Bash 세션 cwd persist

증상. 초기 scaffolding 중 cd .claude/plans/.../signals 한 번 쓴 뒤, 다음 Bash 호출에서 상대 경로가 signals 디렉토리 기준으로 해석되어 logs/.gitkeep 생성 실패.

조치. 이후 모든 오케스트레이터 Bash 호출을 절대 경로 로 고정. 일반 IDE 쉘 가정이 cmux 에서는 깨짐에 유의.

Blocker #5 · 워커 C/D — evaluator_review 신호 전환 시 재프롬프트 (사용자 재개입 필요)

증상 (스크린샷 9.16.17 · 9.21.47 제공). 초기 kickoff 시 "2번(signals/ 항상 허용)" 으로 승인했음에도, 워커가 echo evaluator_review > signals/x.status 로 신호를 바꿀 때 다시 퍼미션 프롬프트 발생. 사용자가 수동 Yes 를 눌러야 워커가 Evaluator dispatch 로 진행.

  • Worker D (ws:13, 약 00:17 UTC): 5 커밋 + push 완료 후 SIGDIR=... echo 1 > d.iteration ; echo evaluator_review > d.status ; ls -la 에서 프롬프트 걸림. 사용자 Yes.
  • Worker C (ws:12, 약 00:22 UTC): 4 커밋 후 echo evaluator_review > c.status ; echo 1 > c.iteration ; cat ... ; git log 에서 프롬프트. 사용자 Yes.

원인 — "always allow" 의 패턴 특이성

Claude Code 의 "2번: always allow for similar commands" 는 경로가 아니라 명령 패턴 (정확히는 첫 인자 + 경로 prefix) 단위로 persist 됨. 즉:

  • 초기 허용: echo running > signals/x.status → "echo running" + "signals/" 조합 persist
  • 이후 새 명령: echo evaluator_review > signals/x.status → 첫 토큰이 running → evaluator_review 로 바뀌면서 새 패턴으로 인식 → 재프롬프트

이는 오케스트레이터 세션에서 나온 스크린샷 9.21.47 의 승인 옵션 "2. Yes, and allow signals/ access and echo evaluator_review and echo 1 commands" 문구에서 직접 확인됨 (이전 "running" 허용과 별개임을 명시).

재발방지 — settings.local.json 에 광역 allow

// .claude/settings.local.json 에 미리 추가하면 신호 전환마다 재프롬프트 없음
"allow": [
  "Bash(echo * > .claude/plans/bcd-orchestration/signals/*)",
  "Bash(echo * > .claude/plans/bcd-orchestration/logs/*)",
  "Bash(tee -a .claude/plans/bcd-orchestration/logs/*)"
]

또는 워커 프롬프트에서 신호 전환 명령을 단일 bash 호출로 묶지 말고 Write 도구로 파일을 덮어쓰게 유도 (Write 는 보통 allow 리스트에 이미 존재) — 하지만 이 경우 로그에 tee 못 하므로 허용 리스트 추가가 현실적.

Blocker #6 · 세션 중간 /btw 메타 질문 — "승인 없이 돌리려면?"

증상 (스크린샷 9.06.45). 실행 중간에 사용자가 /btw 이렇게 중간에 내가 승인을 해줘야했었네 ... 혹시 해당 부분도 없이 그냥 진행이 되기를 원했다면 어떻게 해야됐을까? 로 메타 질문. 이는 병목 자체라기보다 병목의 존재를 사용자에게 드러낸 인지 이벤트.

당시 응답 요지 (스크린샷에 보인 내용 + 본 보고서에 영구 반영)

개입 지점 1 — Phase 0 승인 ("ok"): 설계상 의도된 단일 interaction (원본 명세 line 27-31). 자율 실행의 gate 역할. 제거 대상 아님.

개입 지점 2 — Option A/B/C/D 선택: 진짜 예상 밖. 원인 두 가지 (Blocker #1 + #2 와 동일):

  • (a) --dangerously-skip-permissions 가 claude 2.1.114 부터 --allow-dangerously-skip-permissions gate 를 요구하는 구조 변경을 원본 명세에 반영 안 했음
  • (b) 프로젝트 .claude/settings.json 의 gh pr merge* DENY 를 문서 작성 시점에 사전 검증 안 했음 (문서 line 755 에 실패 케이스만 언급)

핵심 교훈 — 다음 자율 실행 문서 작성 시 사전 검증 5단계

  1. 플래그 현행성: claude --help 로 현재 CLI 의 정확한 bypass 플래그 조합 확인
  2. Trust 다이얼로그: 인터랙티브 cmux 세션은 -p 아니면 반드시 뜸 → cmux send '1\n' 자동화 명시
  3. .claude/settings.json deny/allow 스캔: 워커/오케스트레이터가 쓸 모든 명령군 (echo, pnpm add, git push, gh pr merge 등) 이 deny 에 없는지 · allow 커버되는지 사전 확인. 충돌 발견 시 baseline 을 충돌 없는 방향으로 사전 고정.
  4. hook 조회: settings.json 의 hooks, UserPromptSubmit 등이 중간 개입을 유발하는지
  5. Probe dry-run: 문서 완성 후 1개 단기 스프린트 (30분 이내) 로 실제 kickoff → 머지 파이프 검증. 이걸 건너뛰면 이번처럼 Phase 1 에서 막힘.

요약. "자율 실행 문서" 는 반드시 실제 환경에서 end-to-end probe 를 거쳐야 함. 특히 --dangerously-skip-permissions 처럼 플래그 semantics 가 CLI 버전에 따라 바뀌는 부분과, 프로젝트별 settings.json deny 룰은 문서 작성 시점의 실측이 아니면 예측 못 함. 사전에 플래그 조합 probe + trust 자동화 + settings 충돌 해결책을 baseline 으로 고정했다면 "ok" 한 번 만으로 끝났을 것.

6.1 Git Safety Protocol 의 스피릿 유지

원칙. .claude/settings.json 의 deny 배열은 사용자 의도된 절대 차단 이다. 이번처럼 명시적 한 라인만 임시 제거 → 즉시 복원 패턴은 허용하되, 다음 가드를 추가한다:

  1. 복원 검증 자동화. 머지 직후 grep -n "gh pr merge" .claude/settings.json 로 56번 줄 존재 확인을 Phase 3 의 종료 조건에 포함. (이번 세션에 이미 수행, 앞으로 체크리스트에 표준화)
  2. 우발적 양방향 제거 방지. 임시 제거는 한 줄 단위 로만. 배열 전체 주석/삭제 금지 (JSON 은 주석 불가라는 특성이 오히려 안전판).
  3. LESSON 승격. 본 회고의 LESSON-009 후보 (deny 는 bypass override 불가) 를 .ai/project/lessons.md 영구 규칙으로 승격 검토.

6.2 자율 실행 세션의 권한 범위 — "화이트리스트 우선" 원칙

바이패스 플래그 쌍 (--allow-... --dangerously-...) 은 강력하지만 settings.json 의 allow 확장이 더 안전. 반복 자율 실행이 예상되는 명령군은 settings.local.json allow 리스트에 점진 추가:

// 제안: 자율 워커 친화적 allow 패턴 (.claude/settings.local.json)
"allow": [
  // 워커/오케스트레이터 공통
  "Bash(mkdir *)", "Bash(touch *)", "Bash(tee *)",
  "Bash(pnpm *)", "Bash(uv *)", "Bash(alembic *)",
  "Bash(gh pr create *)", "Bash(gh pr view *)", "Bash(gh run *)",
  "Bash(git push origin feat/*)",  // 기능 브랜치만
  "Bash(git push origin fix/*)",
  "Bash(git push origin chore/*)",

  // 신호 write — "always allow" 의 패턴 특이성 회피 (Blocker #5)
  "Bash(echo * > *.status)",
  "Bash(echo * > *.iteration)",
  "Bash(echo * > *.pr)",
  "Bash(echo * > .claude/plans/*)",  // 모든 신호/플랜 write 광역
  // gh pr merge 는 여전히 deny
]

포인트. echo * 단독으로 열어두면 경로 제약이 사라져 위험. 경로 suffix (.status, .iteration) 로 제한하면 신호 전환 명령 전부 커버하면서 임의 파일 write 는 여전히 prompt. Blocker #5 (Worker C/D 신호 전환 시 재프롬프트) 재발 방지용 핵심 패턴.

이 변경만으로도 앞으로 "트러스트 다이얼로그 자동 수락" 외에 대부분의 bypass 필요성이 사라짐.

6.3 Playwright + Clerk 세션 재사용 (워커 live smoke 안정화)

Sprint C 블로커 재발 방지용 기본 인프라:

// frontend/tests/auth.setup.ts (신규)
import { test as setup } from '@playwright/test';
const AUTH_FILE = 'frontend/.auth/user.json';

setup('authenticate', async ({ page }) => {
  await page.goto('/sign-in');
  // Clerk 테스트 사용자 프로그래매틱 로그인
  await page.fill('input[name=identifier]', process.env.CLERK_TEST_EMAIL!);
  await page.fill('input[name=password]', process.env.CLERK_TEST_PASSWORD!);
  await page.click('button[type=submit]');
  await page.waitForURL('/strategies');
  await page.context().storageState({ path: AUTH_FILE });
});

이후 워커 live smoke 는 storageState: AUTH_FILE 로 시작 → 로그인 우회. .auth/ 는 gitignored. Clerk 테스트 사용자 자격은 secret 으로만 관리.

6.5 Monitor 스톨 감지 + 자동 escalation

이번 세션에서 스톨 감지 Monitor 는 작동했지만 단순 경고만 발생. 다음부터는:

  • 로그 30분+ 정체 → 워크스페이스 스크린 자동 캡처 → 오케스트레이터가 Agent 툴로 screen 해석 → 자동 복구 (send "1" / ESC / 재 kickoff)
  • 연속 2회 stall → 해당 스프린트 blocked 기록 + docs/TODO.md 자동 append

스크립트는 Bash poll 로 작성 가능. 본 세션은 운 좋게 sensitive-file prompt 만 발생했지만, 앞으로 장시간 비감시 실행에서는 필수.

7. 다음 자율 실행 체크리스트

Kickoff 전 (= 자율 실행 문서 작성 시점의 사전 Probe 5단계)

  • ① 플래그 현행성: claude --help | grep permission 으로 현재 CLI 의 정확한 bypass 조합 확인 (2.1.114 기준: --allow-... --dangerously-... 쌍)
  • ② Trust 다이얼로그 자동 수락: 인터랙티브 cmux 는 -p 아니면 반드시 뜸 → sleep 10 && cmux send '1' + Return 명시
  • ③ settings.json deny/allow 스캔: 워커/오케스트레이터가 쓸 모든 명령군 (echo, pnpm, git push, gh pr create/view/merge, mkdir, tee 등) 이 deny 에 없는지 · allow 커버되는지 사전 검증. 신호 전환 (echo running vs echo evaluator_review) 이 별개 패턴임을 인지하고 광역 allow 추가.
  • ④ hook 조회: grep hooks settings.json — UserPromptSubmit 등이 중간 개입 유발하는지
  • ⑤ Probe dry-run: 문서 완성 후 30분 이내 단기 스프린트로 실제 kickoff → 머지까지 end-to-end 검증 (이걸 건너뛰면 이번처럼 Phase 1 에서 막힘)
  • scope 가 baseline 수준까지 고정되었는가? (라이브러리, 상수, scope-out 명시)
  • worker 프롬프트에 질문 금지 + 머지 금지 명시되었는가?

실행 중

  • Monitor 2개 (signal 전환 + log stall) 병렬 가동
  • 워커 스크린 주기적 캡처 (5~10분 간격) — signal stall 없어도 퍼미션 프롬프트에서 대기 가능 (Blocker #5 교훈)
  • 30분 이상 stall 시 screen 캡처 + 원인 분류
  • Evaluator iteration 은 반드시 새 subagent (cold context)

머지 시점

  • deny 라인 임시 제거는 한 줄 단위. 배열 전체 금지
  • 머지 3건 직후 grep 로 deny 라인 복원 확인
  • 머지 순서: CI CLEAN 먼저 → UNSTABLE 대기 → 순차

Cleanup

  • worktree 제거 + local branch 삭제 (git worktree remove --force)
  • cmux 워크스페이스 close
  • orphan 프로세스 (next-server / mcp-chrome) pkill
  • tracker + summary + MEMORY 업데이트
