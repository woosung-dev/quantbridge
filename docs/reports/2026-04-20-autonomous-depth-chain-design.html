autonomous-depth-chain 스킬 설계 (스킬 4)

2026-04-20 · 보완 대상: 스킬 3 autonomous-depth-sprint 의 iteration·배포 확장 · 스킬 2 와의 관계: 스킬 2 가 작은 sprint 의 주간 iteration+배포 였다면, 스킬 4 는 깊은 sprint 의 의존성 체인 iteration+배포

이 문서의 위치: Sprint Pipeline 스킬 생태계의 네 번째 축. 스킬 1 (수평 병렬) × 스킬 2 (수평 iteration+배포) × 스킬 3 (수직 깊이) 에 이어 수직 깊이 × iteration+배포 를 커버. QuantBridge 의 실제 엔진 레이어 확장은 의존성 있는 깊은 sprint 체인 으로 이뤄진다는 관찰에서 출발.

Human Gate 종합

GATE 0: Iteration Plan 승인 — 항상 필수

  • 이유: 2주짜리 commitment + 의존성 DAG 가 틀리면 전체 실패. 사람이 breakdown 을 반드시 검토
  • 확인 항목: N depth sprint 의 scope / 의존성 순서 / 예상 소요 / 롤백 가능성

GATE Sk: 각 Sprint PR MR — N회 필수

  • 이유: 다음 sprint 가 이전 결과 위에 의존하므로 merge 확정 필요. 스킬 1 의 1회 MR 과 다름
  • auto-skip 조건 없음 — Git Safety Protocol 엄수

GATE M: stage→main MR — 필수

  • iteration-X 전체를 main 으로 승급. 2주치 diff 이므로 사용자가 최종 감수

GATE D/N: 배포/다음 루프 — 조건부

  • CLAUDE.md 의 deploy_stage 설정에 따라 자동 skip 가능. staging 전용이면 GATE D 자체 skip

스킬 3 대비 차별점

  • Iteration-level Planner — 단일 sprint 가 아니라 2주치 목표를 N sprint DAG 로 분해
  • Sprint 간 state 주입 — 이전 sprint 의 변경 사항 · lessons · blocker 가 다음 sprint 의 Triage 에 자동 전달
  • Integration Gate N회 — 각 sprint 완료 후 통합 smoke. 누적 regression 조기 차단
  • Iteration Retro + ADR — sprint 단위 ADR 뿐 아니라 iteration 전체를 관통하는 아키 결정 을 별도 문서로 남김
  • Canary 30분 — depth sprint 의 엔진 레이어 변경은 영향 범위 크므로 스킬 2 보다 긴 감시
  • Auto-rollback 지원 — Canary RED 시 자동 reset (단일 sprint 가 아닌 iteration 단위)

스킬 2 와 결합 가능성

미래 확장: 스킬 2 (sprint-cycle) 가 내부적으로 sprint-level 작업을 dispatch 할 때, "이 sprint 는 깊음" 으로 판정되면 스킬 3 대신 스킬 4 의 서브 sprint loop 을 nest 할 수 있음. 즉 스킬 2 가 상위 오케스트레이터, 스킬 4 가 중위, 스킬 3 이 하위. 단 이 3-layer nesting 은 H2 이후 재검토.

스킬 4 가 대체하지 못하는 영역

  • 독립 sprint 3~5 동시 실행 — 스킬 1 이 적합. 스킬 4 는 순차 DAG 전용
  • 단일 깊은 sprint 1개 — 스킬 3 이 적합. 스킬 4 는 최소 2~3 sprint 체인 전제
  • UI 폴리시 월간 누적 배포 — 스킬 2 가 적합. 스킬 4 는 엔진/아키 레이어 대상
  • 프로덕션 장애 대응 — /investigate + systematic-debugging 직접. 스킬 4 는 forward progress 전용

제안 로드맵 (4 스킬 통합)

  1. 현재: 스킬 1 완료, Bundle 2 stage→main 머지 대기
  2. 다음 세션: 스킬 3 프로토타입 수동 실행 — Pine local history ring 또는 Trading WebSocket 중 1개
  3. +1 session: 스킬 3 SKILL.md + scripts 구현 + dogfood
  4. +2 session: 스킬 4 설계 재검토 — 스킬 3 dogfood 경험을 inject 하여 이 문서 업데이트
  5. +3 session: 스킬 4 SKILL.md + scripts 구현 + dogfood (Pine v2 H2 iteration 대상)
  6. H2 진입 후: 스킬 2 (sprint-cycle) 착수 — 이때 스킬 1/4 와 nesting 설계 확정
왜 지금 스킬 4 설계만 해두는가: 스킬 3 을 설계할 때 상위 consumer (스킬 4) 의 interface 를 알면 스킬 3 의 입출력 계약이 더 견고해짐. "내 결과를 다음 sprint 가 context 로 받는다" 는 전제가 있으면 스킬 3 의 Phase 9 memory/ADR 을 더 구조적으로 설계하게 됨. 즉 스킬 4 문서는 스킬 3 의 설계 제약 조건 역할도 한다.
