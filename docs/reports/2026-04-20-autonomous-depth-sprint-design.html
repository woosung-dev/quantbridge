Human Gate 종합

GATE 1: Brainstorming 수렴 — 거의 항상 활성

  • 이유: scope 탐색 자체가 이 스킬의 존재 이유. 사람과 대화하지 않고 단독으로 요구사항 확정하면 scope drift 확정
  • auto-skip 조건: GATE 0 Triage 에서 "명확" 판정 + 유사 theme 완료 sprint 2회↑ 존재 → 바로 Phase 3 로 jump
  • override: raw input 이 plan 파일이면 Phase 4 부터 시작 (Phase 0~3 전체 skip)

GATE 2: Plan 승인 — 필수

  • 이유: 수 시간 실행 전에 사람이 방향 확정. Execute 중 rollback 비용 크므로 사전 gate 불가피
  • 확인 항목: task 분해 / TDD 적용 대상 / 예상 소요 / 리스크 섹션

GATE 3: stage→main MR — 필수 (Git Safety Protocol)

  • 스킬 1/2 와 동일 — Claude 는 main 에 직접 push 금지

GATE 0: Triage (자동 skip 가능)

  • auto-skip 조건: raw input 이 이미 확정 plan 파일 / 유사 sprint 3회+ 성공
  • 강제 활성: DB migration / 신규 envvar / 외부 API / 금융 Golden Rule (AES-256 · Decimal · exec)

스킬 3 이 대체하지 못하는 영역

  • "이거 만들 가치 있나" 단계 — /office-hours 직접 사용 (Phase 1 에서 호출은 하지만, 의사결정은 사람)
  • 병렬 실행 — scope 분해 가능하면 스킬 1 로 이관 (스킬 3 은 1 sprint 전용)
  • 여러 sprint iteration 관리 — 스킬 2 (sprint-cycle) 가 상위에서 orchestrate
  • LLM 도메인 지식 gap — GATE 1/2 둘 다 활성으로도 완벽 아님. 경험 많은 사람 리뷰 필요

제안 로드맵

  1. 즉시: Bundle 2 (stage/fe-polish-b2) A→main 머지 (사용자 터미널 2~3명령)
  2. 이번 세션: 스킬 3 의 프로토타입 수동 실행 1회. 후보 — Pine local history ring 또는 Trading WebSocket. 수동 brainstorming → writing-plans → executing-plans 직렬로 진행하여 반복 패턴 추출
  3. 다음 세션: 프로토타입에서 추출한 패턴으로 스킬 3 SKILL.md + scripts 구현
  4. +1 sprint 후: 스킬 3 dogfood (두 번째 깊이 sprint)
  5. 스킬 3 안정화 후: 스킬 2 (sprint-cycle) 착수
결론: 스킬 2 (sprint-cycle) 를 바로 만드는 것보다 스킬 3 (depth-sprint) 이 이 프로젝트에 더 시급합니다. QuantBridge 는 FE 폴리시보다 Pine/Trading/Backtest 등 엔진 깊이 있는 sprint 가 훨씬 많고, 그 영역이 자동화 없이 수동 진행되면 H2 로드맵 소화 불가능. 스킬 2 는 배포 루프가 확립된 H2 이후로 미뤄도 됨.
