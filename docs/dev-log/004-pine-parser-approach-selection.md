# ADR-004: Pine 파서 접근법 선택 근거

> **상태:** 확정
> **일자:** 2026-04-15
> **출처:** Stage 3 Sprint 1 brainstorming 세션 (spec: `docs/superpowers/specs/2026-04-15-pine-parser-mvp-design.md`)
> **관련:** ADR-003 (Pine 런타임 안전성 + 파서 범위)

---

## 컨텍스트

QuantBridge 파이프라인의 핵심 기술 리스크는 **Pine Script → 실행 가능한 시그널 변환**이다. 이 변환기를 어떤 아키텍처로 구현할 것인지 결정이 필요했다. 선택이 뒤집히면 실행 엔진, 편집기, 백테스트 엔진, 장기 확장 기능(AI 변형, 전략 병합) 전부가 영향을 받는다.

---

## 검토한 3가지 접근법

### 접근 1 — AST 인터프리터
`Pine → 자체 AST → Python 비지터로 순회 실행.` 지원 함수 화이트리스트를 AST 노드에서 검증.

### 접근 2 — Python 소스 트랜스파일
`Pine → Python 코드 문자열 생성 → RestrictedPython 샌드박스에서 exec 실행.`

### 접근 3 — 시그널 DSL 직매핑
`Pine → 구조화 객체(전략 패턴) → vectorbt 시그널 함수 호출로 직접 매핑.` 실행 로직 없음.

---

## 비교 매트릭스

| 항목 | 접근 1 (AST) | 접근 2 (트랜스파일) | 접근 3 (DSL) |
|------|--------------|---------------------|--------------|
| ADR-003 보안 (exec/eval 금지) | ★★★ 만족 | ★★ RestrictedPython 탈출 리스크 | ★★★ 만족 |
| 정확도 (stdlib 검증 라이브러리 래퍼 전제) | ★★★ | ★★★ | ★★★ |
| 표현력 (if/for/커스텀 함수/var state) | ★★★ | ★★★ | ★ 정형 패턴만 |
| 확장성 (새 문법 추가) | ★★★ 비지터 메서드 추가 | ★★★ 트랜스파일 규칙 추가 | ★ 매핑 테이블 폭발 |
| 속도 (일회성 백테스트) | ★★ | ★★ | ★★★ vectorbt 네이티브 |
| 속도 (최적화 그리드) | ★★ (Numba 가속 가능) | ★★★ 컴파일 후 N번 재실행 | ★★★ |
| 구현 비용 | 중~높음 | 중간 | 낮음 (단 확장 시 폭발) |
| 편집/변형/병합 적합성 | ★★★ AST는 편집 표준 | ★ Python 편도 차선 | ★★ 레고식 조합만 |
| Pine 역출력 (왕복) | ★★★ pretty printer | ★ 양방향 변환 지옥 | ★ 패턴→템플릿만 |

## 실제 TV 스크립트 5개 표본 검증

| # | 스크립트 | 특징 | 접근 1 | 접근 2 | 접근 3 |
|---|----------|------|--------|--------|--------|
| 1 | DrFX Diamond Algo (v5) | 커스텀 함수 20+개, var state, MTF, 드로잉 집약, 배열 기반 공급/수요 | ✅ 장기 가능 | ✅ 장기 가능 | ❌ |
| 2 | Moon Phases Strategy (v4) | 커스텀 수식(moon cycle), var bool, if/else if/else, valuewhen | ✅ | ✅ | ❌ 커스텀 수식 매핑 불가 |
| 3 | SuperTrend Strategy (v4) | 표준 지표, `:=` self-reference, 시간 윈도우 필터 | ✅ | ✅ | ⚠️ 변형 없을 때만 |
| 4 | Flawless Victory (v4) | v1/v2/v3 분기, `strategy.exit(stop, limit)`, position_avg_price | ✅ | ✅ | ❌ 다중 버전 + 브래킷 오더 |
| 5 | EMA Cross (v4) | EMA + crossover + 시간 윈도우 | ✅ | ✅ | ✅ |

**결과:** 접근 1만 5개 모두 일관되게 커버. 접근 3은 5개 중 1개만 깔끔 지원. 접근 2는 접근 1 대비 유니크한 이점 없음 + 보안 비용.

---

## 결정: 접근 1 (AST 인터프리터)

### 근거

1. **ADR-003 부합도 최상.** `exec`/`eval` 없이 비지터 패턴으로 실행. 샌드박스 탈출 리스크 0.
2. **TV 실제 스크립트 5개 표본에서 유일하게 모두 커버 가능.** 접근 3은 극히 단순한 전략만, 접근 2는 유니크 이점 없음.
3. **AST는 실행 외 장기 확장의 중앙 저장소 역할을 겸한다.**
   - AST → Python 읽기 전용 렌더러 (학습/이식 용도)
   - AST → Pine pretty printer (편집 후 역출력)
   - AST 변형 (AI 지표 추가/교체)
   - 여러 전략 AST 병합/차분 (조합 전략)
   - 초보자 카탈로그 빌더가 AST를 생성·수정
4. **비대칭 흡수 가능성.** 접근 1 기반이면 필요 시 접근 3 스타일 패스트패스(SuperTrend 같은 표준 패턴 감지 → vectorbt 직호출 우회)를 나중에 얹을 수 있음. 접근 3 기반에서 접근 1 능력을 추가하려면 기반부터 다시 써야 함.

---

## 접근 2가 탈락한 상세 근거

### 유니크한 강점 부재

| 주장되는 접근 2의 이점 | 실제 검증 |
|------------------------|-----------|
| 대량 백테스트 속도 | 접근 1 + Numba/vectorbt로 동등 가능 |
| 생성 Python을 사람이 읽을 수 있어 투명성 | **접근 2 실행이 아니라 "AST → Python 읽기 전용 렌더러"로 분리 구현 가능** (§"대안 흡수" 참조) |
| 사용자가 Python으로 직접 편집 | 우리 제품은 Pine 플랫폼, Python 편집은 동기화 악몽 |
| AI가 Python을 잘 이해 | AST → Python 렌더러로 동등 효과 |

### 치명적 단점

- **ADR-003 위반 여지.** RestrictedPython은 과거 CVE 이력(탈출 취약점)이 있어 "안전"을 단언 불가. 보안 리뷰 비용이 영구 부담.
- **에러 추적 지옥.** 런타임 에러가 생성된 Python 코드의 라인 N에서 터지면 원본 Pine의 어느 줄인지 역매핑 필요 → 소스맵 별도 구현.
- **편도 차선.** Python으로 번역되는 순간 Pine 구조 정보 손실. 우리 제품의 "TV 전략 고도화 허브" 비전(Pine↔편집↔Pine 왕복)과 충돌.
- **유지보수 이중화.** 파서 + 트랜스파일 규칙 테이블을 둘 다 유지. 접근 1 대비 유지보수 면적이 더 크지만 산출물은 본질적으로 동등.

### 결론
**"속도 이점"은 다른 경로로 달성 가능하고, "Python 가시성"은 읽기 전용 렌더러로 분리 흡수 가능.** 접근 2만의 유니크한 가치가 없는 상태에서 ADR-003 보안 부담을 떠안을 이유가 없음. **영구 불채택.**

---

## 접근 3가 탈락한 상세 근거

### 표현력 한계

접근 3의 작동 방식은 "사전 정의된 전략 패턴 카탈로그에서 매칭"이다. 즉 다음과 같은 Pine 구문이 등장하면 바로 깨진다:

- **커스텀 수식** (예: Moon Phases의 `(new + time + day*2)%cycle/cycle`)
- **커스텀 함수 정의** (예: DrFX Diamond의 `supertrend()`, `dirmov()` 등 20+개)
- **조건 조합** (예: "크로스오버 AND 볼륨 필터 AND RSI < 70") — 조합마다 패턴 등록 필요
- **var state + `:=` 재할당** (bar 간 상태 누적)
- **`strategy.exit(stop, limit)` + `strategy.position_avg_price`** 같은 동적 파생값
- **v1/v2/v3 버전 스위치** (같은 스크립트 안에서 다중 전략 분기)

### 실제 데이터

5개 표본 중 "깔끔하게 가능"한 건 1개 (EMA Cross). 나머지 4개는 전부 어딘가에서 블로커 발생.

### 잔존 가치

접근 3은 "초보자용 드롭다운 빌더 UX"로는 여전히 가치가 있음. 단 이건 실행 엔진이 아니라 UI 레이어이며, **접근 1의 AST 위에 얹어도 동일하게 구현 가능**. 즉 실행 엔진으로 접근 3을 채택할 이유는 없다.

### 결론
**실행 엔진으로는 불채택.** 단 Phase 2~3의 초보자 빌더 UX 레이어와 (측정 시) Optimizer 성능 패스트패스 후보로는 보류.

---

## 대안 흡수: 접근 1 기반으로 타 접근의 강점 취하기

접근 1을 베이스로 두면 필요 시 접근 2·3의 유용한 측면을 비침습적으로 얹을 수 있다:

| 흡수할 가치 | 구현 방식 | 타이밍 |
|-------------|-----------|--------|
| 접근 2의 "Python 가시성" | AST → Python 읽기 전용 렌더러 (실행 X) | Phase 2~3 |
| 접근 3의 "표준 패턴 UX" | AST를 생성·수정하는 드롭다운 빌더 | Phase 3 |
| 접근 3의 "속도" | AST 분석으로 표준 패턴 감지 → vectorbt 직호출 패스트패스 | 성능 병목 실측 후 |
| 접근 2의 "컴파일 후 재실행" 속도 | 접근 1 + Numba 가속 | 최적화 스프린트 |

---

## AST의 위상 — 제품 차원에서

AST는 실행을 위한 중간 자료구조를 넘어서 **제품의 "중앙 편집 가능 표현"**이다. QuantBridge가 "TV 전략을 가져다 실행"하는 도구가 아니라 **"TV 전략을 고도화하고 조합하는 허브"**가 되려면 편집 가능한 구조화 표현이 필수. AST가 그 역할을 맡고, 모든 장기 확장(AI 변형, 전략 병합, Python/Pine 렌더링, 빌더 UX)이 같은 AST를 공유한다.

이 결정은 AST 노드 설계에도 영향을 준다:
- `@dataclass(frozen=True)` — 불변 (안전한 변환)
- `source_span` 포함 — 에러 추적 + 역출력 + 원본 라인 주석 삽입
- `annotations: dict` 슬롯 — 향후 AI 태깅, 전략 병합 메타데이터

| 선택 | 결정 |
|------|------|
| 실행 엔진 접근법 | **접근 1 (AST 인터프리터)** |
| 접근 2 (실행 목적) | **영구 불채택** |
| 접근 2의 "Python 가시성" | AST → Python 읽기 전용 렌더러로 분리 흡수 (Phase 2~3) |

| 접근 3 패스트패스 | 성능 병목 실측 시에만 조건부 검토 |
